文档概述

本文档描述 ActionDecisionModule（ADM） 的完整设计。ADM 是连接高层规划与底层执行的核心模块，负责将 PlanningModule 输出的自然语言步骤拆解为可执行的原子动作序列，并处理执行过程中的动态事件（威胁、路径冲突、队友状态变化）触发的重规划。

系统分层

层级模块职责状态
用户输入SubmitMissionRequest自然语言任务描述已有
规划层PlanningModule任务 → 分组 → 步骤分配已有
决策层ActionDecisionModule步骤 → 原子动作 → 执行 + 重规划本文档
执行层NavMesh / A* / 感知模块底层导航与感知已有
01

模块定位与接口

与 PlanningModule 的边界

PlanningModule（LLM4）输出自然语言步骤数组，例如：

// LLM4 输出的 PlanStep
{ "stepId": "step_1", "text": "飞往A楼",       "constraint": "" }
{ "stepId": "step_2", "text": "护送目标到南门", "constraint": "保持前探-侧护-后卫队形" }
⚠

关键原则：ADM 是每个智能体独立拥有的模块，重规划由智能体自主触发，不依赖组长仲裁。组长角色仅在 PlanningModule 分组协商阶段有效，进入执行阶段后所有智能体完全对等。

ADM 对外接口

方法调用方说明
StartStep(PlanStep)PlanningModule传入新步骤，触发 LLM-A 解析
OnPerceptionEvent(event)感知模块感知事件输入，可能触发重规划
OnBoardUpdate(update)CommunicationModule收到队友黑板更新
GetCurrentAction()执行层获取当前原子动作
CompleteCurrentAction()执行层通知当前动作完成，推进队列
02

地图表示方案

LLM 在拆解原子动作时需要理解地图，以便推断中间途径节点、判断威胁相对方向、选择绕行路线。核心约束：LLM 无法从坐标数字推断空间关系，必须提供人类可读的 2D 空间描述；同时校园节点数量可能达 60 个以上，不能全量传入。

方案：动态子图裁剪（基于 SayPlan 思路）

SayPlan 的核心价值在于让 LLM 只关注任务相关的子图，而不是暴露整张地图。对于本系统，由于大节点只有名称和网格位置（无内部细节），无需"展开层"。替代方案是：

  • 全局折叠图（始终传入，约 150 token）：ASCII 网格，覆盖全校园，仅显示节点名称和类型标注
  • 任务子图（动态裁剪，约 50–100 token）：根据步骤起点和终点，裁剪出路径走廊的局部子网格，作为补充传入
ℹ

裁剪策略：以起点和终点的网格坐标为对角，扩展 1–2 格 padding，形成包围矩形，只传该矩形内的节点。这样 LLM 既能看到全局方向感，又能在局部子图中精确推断途径节点。

全局折叠图示例

校园地图概览（每格约50m）：

      [西]          [中]          [东]
[北]  北区树林 #    A楼 ·         停车场B □
      操场 ▽         中心湖 *      B楼 ·
[南]  图书馆 ·      南门 =         运动场 ▽

图例：· 建筑  = 道路  ▽ 运动/绿地  # 林地  □ 停车  * 禁飞水域
相邻格可直达，禁飞区(*)不可进入，林地(#)可通行但提供遮蔽

任务子图示例（步骤：从 A楼 护送到 南门）

任务相关子图（路径走廊）：

      [西]        [中]        [东]
[北]  A楼 ·       停车场B □   ...
[中]  操场 ▽      中心湖 *    B楼 ·
[南]  图书馆 ·    南门 =      运动场 ▽

注：停车场B方向已有威胁报告（Agent_02广播），建议绕行
可选路径：A楼 → 操场 → 图书馆 → 南门（西侧绕行）
          A楼 → B楼 → 南门（东侧绕行，需确认B楼通行状态）
✓

验证方法：生成地图后，直接询问 LLM "从图书馆到停车场B可以经过哪些节点"，检验推理是否正确。若推理错误，调小格子尺寸提升精度（默认 10×8，上限 12×10）。

自动生成方式

折叠图和子图均由代码从 CampusGrid2D 自动生成：

  1. 取所有大节点的 featureSpatialProfileByUid，提取 worldCenter 坐标
  2. 以地图包围盒为基准，将坐标量化为网格索引（默认 10×8）
  3. 同格多节点时按优先级保留（Building > Road > Sports > Green > 其他）
  4. 禁飞节点（Water）加 * 标注，林地加 # 标注
  5. 任务子图：以步骤起终点网格坐标为对角，扩展 2 格 padding，裁剪矩形区域
  6. 若黑板上有威胁 event，在子图注释中附加威胁位置提示

节点幻觉防护（参考 SayPlan 迭代重规划）

LLM 输出的 targetName 可能是不存在于地图的幻觉节点。执行前验证：若节点不存在于 featureAliasNameMap，将反馈返回 LLM 重新选择，最多重试 2 次，仍失败则标记步骤 Failed。

核心数据结构

原子动作（AtomicAction）

[Serializable]
public class AtomicAction
{
    public string actionId;        // "aa_1", "aa_2"
    public AtomicActionType type;   // 枚举，见下表
    public string targetName;      // 目标语义名称，如"A楼"，底层查 Grid 解析坐标
    public string targetAgentId;   // 目标是智能体时使用（跟随/护送）
    public float  radius;          // 巡逻半径、观察距离等
    public float  duration;        // 等待时长（秒），-1 表示条件触发结束
    public string broadcastContent;// Broadcast 动作的内容
}

public enum AtomicActionType
{
    MoveTo,        // 移动到目标节点
    PatrolAround,  // 绕目标节点巡逻
    Observe,       // 原地观察，激活感知
    Wait,          // 等待（时长或条件）
    FormationHold, // 保持队形偏移跟随目标智能体（具体偏移由协同约束描述）
    Broadcast,     // 广播消息到组内黑板
    Evade,         // 机动规避，目标由重规划给出
}
ℹ

队形角色和偏移距离不作为原子动作字段。这类约束属于协同语义层，由 coordinationConstraint 字段传递，LLM 在生成原子动作时将其体现在 FormationHold 动作的 targetAgentId 和执行逻辑中。

执行上下文（ActionExecutionContext）

执行上下文是重规划能够正确进行的核心依据。举例：智能体正在执行步骤"护送目标到南门"，已拆解出 3 个原子动作：

原子动作状态
aa_1: MoveTo(操场)已完成
aa_2: MoveTo(A楼)执行中
aa_3: MoveTo(南门)待执行

此时感知到威胁，触发重规划。LLM-B 需要知道：当前步骤的原始意图是什么、我现在在哪里、最终目标是哪里、还有哪些节点没走、需要遵守什么协同约束。这些信息全部存在执行上下文中，使 LLM 不必从头重规划整个步骤，只需规划从当前位置到终点的剩余路径。

[Serializable]
public class ActionExecutionContext
{
    // 任务来源
    public string   msnId;
    public string   stepId;
    public string   stepText;               // 原始步骤自然语言，重规划时传给 LLM
    public string   coordinationConstraint; // 从 slot 继承的协同约束
    public RoleType role;

    // 执行状态
    public AtomicAction[] actionQueue;
    public int           currentActionIdx;
    public ADMStatus     status;

    // 空间状态（重规划时告知 LLM 当前处境）
    public string   currentLocationName; // 当前所在区域语义名称
    public string   originalGoalName;   // 步骤级别的最终目标节点
    public string[] remainingWaypoints;  // 尚未到达的途径节点
}

黑板条目（AgentContextUpdate）

黑板是组内共享的情境记忆，每个智能体维护一份 Dictionary<agentId, AgentContextUpdate>，收到队友广播即更新对应条目。

[Serializable]
public class AgentContextUpdate
{
    public string   agentId;

    // 当前状态（结构化，每次原子动作切换时更新）
    public string   locationName;   // 当前语义位置
    public string   currentAction;  // AtomicActionType 枚举名
    public string   currentTarget;  // 当前动作目标节点名
    public string   role;           // 当前角色

    // 未来意图（LLM-A/LLM-B 完成后广播，重规划后更新）
    public string[] plannedTargets; // 计划路径节点序列，如 ["操场","A楼","南门"]

    // 事件记录（最近3条，时间倒序；保留时序便于 LLM 推断事件先后）
    public string[] recentEvents;

    public float    timestamp;
}

协同约束的传递路径

阶段模块操作
LLM1PlanningModule识别用户任务中的协同约束关键词
LLM2PlanningModule提取约束写入 slot.coordinationConstraint
LLM4PlanningModule步骤继承约束，写入 step.constraint
LLM-AADM约束作为 prompt 输入，指导原子动作序列生成
LLM-BADM重规划时约束同样传入，确保调整后仍满足约束

LLM-B：重规划

属性值
触发时机感知事件或黑板 event 更新后，智能体自主判断需要重规划
输入执行上下文 + 对话式组内情境（含所有人 plannedTargets）+ 地图 + 协同约束
输出新的 AtomicAction[] JSON，替换 actionQueue 中剩余部分
关键差异输入包含所有队友的 plannedTargets，确保新计划不产生新冲突
LLM-B 对话式情境格式（参考 RoCo）
组内当前情境： Agent_02（前探）：位置=停车场B附近，动作=MoveTo→南门 计划路径：[停车场B, B楼, 南门] 近期事件：["停车场B发现敌方无人机，已停止移动"] Agent_03（后卫）：位置=图书馆，动作=FormationHold→Target_01 计划路径：[图书馆, 南门] 近期事件：[] 我（Agent_01，侧护）：当前位置=A楼，已完成 MoveTo(操场)→MoveTo(A楼) 原计划路径：[操场, A楼, 停车场B, 南门] 剩余途径点：[停车场B, 南门] 中断原因：停车场B发现敌方无人机 原始步骤：护送目标到南门 协同约束：保持前探-侧护-后卫队形 [全局折叠图 + 任务子图] 请重新规划我的剩余原子动作，不得与任何队友路径冲突，并维持协同约束。 输出纯 JSON 数组。
05

冲突检测与解决流程

无论是执行前初始协商，还是执行中动态重规划，均走同一套统一流程。

完整流程

  1. LLM-A 或 LLM-B 完成，生成 plannedTargets 候选
  2. 广播 AgentContextUpdate（含新 plannedTargets）到组内黑板
  3. 每个智能体代码层检测：将自身 plannedTargets 与黑板所有条目做交集
  4. 统计各冲突方涉及的冲突节点数，冲突节点最多的那个成为解决者（节点数相同取 agentId 字典序最小者）
  5. 解决者调用 LLM-B，输入：自身计划 + 所有人 plannedTargets + 地图 + 协同约束
  6. 解决者输出新的原子动作序列（已解决所有冲突），广播新 plannedTargets
  7. 其余冲突方代码验证：确认新 plannedTargets 与自身无冲突，则接受并开始执行
  8. 全员开始或继续执行
⚠

核心设计原则：一次 LLM-B 调用解决所有冲突。解决者在输入中收到所有人的 plannedTargets（包括无冲突的队友），输出的新计划必须与所有人兼容，不存在多轮协商链。

中断判断逻辑

收到感知事件或队友 event 广播后，进入 Interrupted 状态，代码层判断是否需要重规划：

条件判断处理
事件位置在 remainingWaypoints 路径上或附近需要重规划触发 LLM-B，走冲突解决流程
协同约束因事件失效需要重规划触发 LLM-B，走冲突解决流程
事件与自身路径无交集不需重规划更新黑板 recentEvents，继续执行

ADM 状态机

状态触发条件说明
Idle初始 / 步骤完成后等待新步骤
Interpreting收到新 PlanStepLLM-A 调用中，等待原子动作序列
NegotiatingLLM-A 完成广播 plannedTargets，等待冲突检测窗口（约 0.5s）
Running无冲突 / 冲突已解决逐条执行 AtomicAction
Interrupted感知事件 / 黑板 event 触发暂停执行，判断是否需要重规划
Replanning判断需要重规划LLM-B 调用中
Done所有动作完成通知 PlanningModule.CompleteCurrentStep()
Failed重规划失败 / 超时通知 PlanningModule，等待降级处理

状态转移图

收到 PlanStep Idle ──────────────────────► Interpreting (LLM-A) ▲ │ │ CompleteCurrentStep │ 解析完成 │ ▼ Done ◄──────────────────── Negotiating (广播 plannedTargets) ▲ 所有动作完成 │ │ │ 无冲突 / 冲突已解决 │ ▼ └──────────────────────── Running (逐条执行) │ 感知/黑板事件│ ▼ Interrupted (判断是否重规划) │ │ 不需重规划 │ │ 需要重规划 ▼ ▼ Running Replanning (LLM-B) │ │ 完成 → Negotiating ▼ Negotiating

通信设计

新增 MessageType

在现有 CommunicationModule 的 MessageType 枚举基础上，ADM 层面仅新增一个消息类型：

// 新增到 MessageType 枚举
BoardUpdate,  // ADM 广播黑板更新，payload 即 AgentContextUpdate

所有 ADM 层面的通信（状态同步、事件广播、冲突协商）均通过此单一消息类型承载，不新增其他消息类型。

广播时机

时机更新字段作用
LLM-A / LLM-B 完成后plannedTargets触发队友冲突检测窗口
原子动作切换时currentAction, currentTarget, locationName保持黑板实时性
感知到新事件时recentEvents（prepend，保留最近3条）通知队友，触发其中断判断

收到 BoardUpdate 的处理逻辑

  1. 更新本地 Dictionary<agentId, AgentContextUpdate>
  2. 若当前状态为 Running：检测对方新 plannedTargets 是否与自己有交集，有则进入 Interrupted
  3. 若当前状态为 Interrupted / Replanning：仅更新黑板，不打断当前重规划流程
  4. 若对方 recentEvents 有新事件：代码判断事件是否影响自身路径，决定是否进入 Interrupted
08

与 PlanningModule 集成

PlanningModule 需要的最小改动

现有代码只需两处修改：

// 1. PlanSlot 新增字段
public string coordinationConstraint;
// 示例："不同方向接近目标" / "保持前探-侧护-后卫队形，间距15m"

// 2. PlanStep 新增字段
public string constraint;
// 从 slot.coordinationConstraint 继承，ADM 拆原子动作时使用

LLM2 的 prompt 新增一句提取指令："若任务描述中包含协同约束（如路径分工、队形、同时到达等），提取并写入 coordinationConstraint 字段；否则留空。"

完整调用链

用户输入 → PlanningModule.SubmitMissionRequest() └→ LLM1 解析任务（relType, groupCnt, groupMsns） └→ 分组 + 广播 GroupBootstrap └→ LLM2 生成 slot（含 coordinationConstraint） └→ LLM3 选槽 └→ LLM4 拆步骤（step 继承 constraint） └→ SetState(Active) → ADM.StartStep(step) └→ LLM-A 拆原子动作 └→ 广播 plannedTargets → Negotiating └→ 冲突检测 → 解决者调 LLM-B（若有冲突） └→ Running：逐条执行 AtomicAction └→ [感知事件] → Interrupted → 判断 └→ 需重规划 → LLM-B → 广播 → Negotiating → Running └→ 所有动作完成 → Done └→ PlanningModule.CompleteCurrentStep() └→ 下一步 → ADM.StartStep(nextStep)
