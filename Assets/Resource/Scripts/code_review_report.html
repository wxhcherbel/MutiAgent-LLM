核心差异化定位

UnityLLM 是唯一将 LLM 多智能体协调与物理仿真执行层深度绑定的框架。MetaGPT 等框架在「纯文本/代码执行」层面协作，而 UnityLLM 需要处理空间推理、物理约束、实时运动控制，这是一个显著更难但也更有价值的研究方向。

1.2 方法论评价 — 4阶段LLM规划流水线

流水线总览

阶段功能对应经典方法评价
LLM#1 自然语言任务解析 → ParsedMission HTN 任务分解 / STRIPS 目标识别 设计合理 解决了非结构化输入到结构化规划的桥梁问题
LLM#2 组长生成计划槽（角色+任务分配） MARL 角色分配 / Contract Net Protocol 有风险 LLM生成质量不稳定，缺乏验证

LLM#4 槽位拆解为有序步骤 HTN Method Decomposition 设计合理 约束绑定到步骤的设计较好
与经典规划方法的深度对比 ▶

vs HTN（层次任务网络）

HTN 通过预定义的 Method 库进行分解，确定性强但灵活性差。UnityLLM 用 LLM 替代 Method 库，获得了自然语言理解能力，但失去了确定性保证。这是一个有意义的权衡——对于开放域任务，预定义 Method 库不现实。

vs STRIPS/PDDL

经典规划需要完整的 domain 和 problem 定义。UnityLLM 完全由 LLM 从自然语言推导，省去了形式化建模的人工成本，但也放弃了正确性验证。建议：在 LLM 输出后加入轻量 PDDL 验证层。

vs MARL（多智能体强化学习）

MARL 需要大量训练数据和环境交互。UnityLLM 用 LLM 的 zero-shot 能力替代训练过程，适合数据稀缺场景，但缺乏 MARL 的在线优化能力。

vs BDI（信念-欲望-意图）

BDI 架构中的 Belief/Desire/Intention 映射到 UnityLLM 中：MemoryModule ≈ Belief, PlanSlot ≈ Desire, AgentPlan.steps ≈ Intention。设计直觉是正确的，但缺少显式的 Intention 修正机制（BDI 的核心优势是动态重规划）。

约束体系总览

约束类型 语义 对应经典理论 实现 评价
C1 资源分配 subject→targetObject 独占绑定 互斥锁 / 资源分配问题 LLM#2 静态分配 OK 简单有效

与经典协调理论的详细对比 ▶

vs Blackboard Architecture（黑板架构）

SharedWhiteboard 是经典黑板架构的简化实现。经典黑板有 知识源(KS)、触发条件、调度器 三要素。UnityLLM 的白板只有读写接口，缺少触发条件（需要 ADM 主动轮询）和调度器（无优先级排序）。

vs Contract Net Protocol

LLM#2→LLM#3 的槽广播→选择→确认流程，本质上是一个简化版的 Contract Net：组长发布 Task Announcement（SlotBroadcast），成员提交 Bid（SlotSelect），组长做 Award（SlotConfirm）。但缺少经典 CNP 的多轮竞价和拒绝机制。

vs Stigmergy（间接协调）

C3 sign=-1 的互斥机制（通过白板 IntentAnnounce 标记已占目标）本质是一种 Stigmergy 协调——Agent 通过环境间接通信。这种设计在无人机群体中是非常自然且合理的，但需要解决并发写入的原子性问题。

1.4 研究创新点识别

创新点 1: LLM驱动的结构化约束提取

从自然语言任务描述中自动识别 C1/C2/C3 三类协调约束，无需人工建模。这在现有文献中少见——大多数 LLM multi-agent 框架的协调规则是硬编码的。

高创新性

创新点 2: 约束感知的滚动执行

ADM 的 RunRollingLoop 在每轮决策时将白板状态（occupied/signals/intents）注入 LLM 提示词，实现约束感知的在线规划。这比一次性规划更具鲁棒性。

高创新性

创新点 3: 物理仿真闭环验证

通过 Unity3D 物理引擎进行实际空间运动仿真，LLM 的决策需要经过物理世界检验。这比纯文本模拟更有说服力，可以暴露空间推理错误。

高创新性

创新点 4: 多关系类型支持

原生支持 Cooperation/Competition/Adversarial/Mixed 四种关系类型，约束按 groupScope 隔离。现有 LLM 框架几乎不支持对抗场景。

1.6 关键风险

风险 1: LLM 输出不确定性导致系统级故障

严重程度: 高 — LLM 的 JSON 输出格式不可靠是最大系统风险。当前代码中 LLM#1~#4 的每一步都依赖 JSON 解析，任何一步失败都会导致整个规划流水线中断。

  • PlanningModule.cs:194 — LLM#1 解析失败直接 Failed
  • PlanningModule.cs:318 — LLM#2 解析失败直接 Failed
  • ActionDecisionModule.cs:244 — ADM 滚动规划解析失败直接 Failed

缺少重试、降级、部分解析恢复等容错机制。

风险 2: 白板无原子性 → C3 互斥竞态条件

严重程度: 中高 — C3 sign=-1 的互斥机制基于「先写入 IntentAnnounce 的 Agent 占有目标」，但两个 Agent 可能在同一帧内同时写入，导致两者都认为自己成功占有。

虽然 Unity 是单线程的，但协程的 yield 点可能导致交错执行。

风险 3: 无端到端可运行的测试验证

严重程度: 高 — 系统有大量组件但缺少集成测试。整个 LLM#1→LLM#2→LLM#3→LLM#4→ADM→AME 的流水线从未被完整验证过，各组件间的数据格式假设可能不匹配。

风险 4: Prompt 脆弱性

严重程度: 中 — 提示词中包含大量格式要求和示例，但 LLM 对长 prompt 中的细节指令遵从率会下降。LLM#2 的 prompt 已超过 1000 字，包含 17 条规则，LLM 很难全部遵守。

风险 5: 超时机制不完善

严重程度: 中 — CheckTimeout() 仅在中间状态触发 25s 超时。但 ADM 的 WaitForC2Sync 使用独立 30s 超时，两个超时机制不协调，可能导致规划层已 Failed 但执行层还在等待。

2.1 LLM调用可靠性

关键 JSON输出验证与重试机制 ▶

问题描述

所有 LLM 调用（LLM#1~#4 + ADM RollingPlan）只做了一次 try-catch，失败后直接中断。无 JSON Schema 验证，无重试，无降级。

受影响文件

  • PlanningModule.cs:192-202 — LLM#1
  • PlanningModule.cs:297-322 — LLM#2
  • ActionDecisionModule.cs:239-257 — ADM Rolling

改善方案

引入「重试+Schema验证+降级」三层防护：

// 建议新增: LLMCallHelper.cs
public static class LLMCallHelper
{
    public static IEnumerator CallWithRetry<T>(
        LLMInterface llm,
        string prompt,
        Action<T> onSuccess,
        Action<string> onFail,
        int maxRetries = 2,
        int maxTokens = 800,
        float temperature = 0.3f) where T : class
    {
        for (int attempt = 0; attempt <= maxRetries; attempt++)
        {
            string result = null;
            yield return llm.SendRequest(prompt, r => result = r,
                temperature: temperature, maxTokens: maxTokens);

            if (string.IsNullOrWhiteSpace(result)) continue;

            try
            {
                string json = ExtractJson(result);
                T parsed = JsonConvert.DeserializeObject<T>(json);
                if (parsed != null)
                {
                    onSuccess(parsed);
                    yield break;
                }
            }
            catch (Exception e)
            {
                Debug.LogWarning($"[LLM] Attempt {attempt+1} parse failed: {e.Message}");
                // 重试时在 prompt 末尾追加纠错提示
                if (attempt < maxRetries)
                    prompt += $"\n\n[上次输出JSON解析失败:{e.Message}。请严格输出合法JSON。]";
            }
        }
        onFail?.Invoke("All retries exhausted");
    }
}

2.2 协调协议鲁棒性

关键 C3互斥竞态 + C2超时处理 ▶

问题 A: C3 sign=-1 竞态条件

两个 Agent 在同一 Unity 帧内同时对同一目标写入 IntentAnnounce，都会认为自己是先到者。

改善方案: 引入序号仲裁

// SharedWhiteboard.cs 中新增原子性写入
private int _globalSeqCounter;

public WriteResult TryClaimExclusive(string groupId, WhiteboardEntry entry)
{
    entry.timestamp = Time.time;
    int seq = ++_globalSeqCounter;

    if (!_store.TryGetValue(groupId, out var list))
    {
        list = new List<WhiteboardEntry>();
        _store[groupId] = list;
    }

    // 检查是否已有其他 Agent 对同一 constraintId 写入了占位
    var existing = list.Find(e =>
        e.constraintId == entry.constraintId &&
        e.entryType == WhiteboardEntryType.IntentAnnounce &&
        e.agentId != entry.agentId &&
        e.status == 1);

    if (existing != null)
        return new WriteResult { success = false, occupiedBy = existing.agentId };

    // 无冲突,写入
    WriteEntry(groupId, entry);
    return new WriteResult { success = true, sequence = seq };
}

2.3 记忆检索效率

重要 文本相似度计算 + 记忆容量管理 ▶

问题描述

MemoryModule.cs:675-696 中的 ComputeTextSimilarity 使用简单的 substring contains 匹配，这在中文场景下效果较差（中文分词问题）。且每次 Recall 都遍历全部记忆做打分（O(n)），当记忆量增大时性能会下降。

当前实现

    score += token.Length >= 4 ? 0.9f : 0.45f;

改善方案

1) 为中文场景引入 n-gram 或字符级 Jaccard 相似度；2) 添加倒排索引加速检索：

// 倒排索引: token → 包含该 token 的 memory id 集合

    var candidates = memories.Where(m => candidateIds.Contains(m.id));
    // ... rest of scoring ...

2.5 通信容错

重要 消息丢失与重复处理 ▶

问题描述

CommunicationManager.cs:64 — 非 reliable 消息有随机延迟但无丢包模拟；reliable 消息直接投递无确认机制。一旦消息在 ResolveRecipients 阶段找不到目标（Agent 尚未注册），消息静默丢失。

关键场景：GroupBootstrap 消息在 Agent 尚未完成 Start() 时发出，接收方 CommunicationModule 未注册，消息丢失 → 该 Agent 永远无法加入组。

改善方案

// CommunicationManager.cs — 为 reliable 消息添加重试队列
private readonly Queue<(AgentMessage msg, float nextRetry, int retries)> _pendingReliable = new();

public void ProcessMessage(AgentMessage message)
{
    // ... existing ...
    if (message.Reliable)
    {
        var recipients = ResolveRecipients(message);
        if (recipients.Count == 0)
        {
            _pendingReliable.Enqueue((message, Time.time + 0.5f, 0));
            return;
        }
    }
    // ... existing delivery ...
}

private void Update()
{
    // 重试挂起的 reliable 消息
    int count = _pendingReliable.Count;
    for (int i = 0; i < count; i++)
    {
        var (msg, nextRetry, retries) = _pendingReliable.Dequeue();
        if (Time.time < nextRetry) { _pendingReliable.Enqueue((msg, nextRetry, retries)); continue; }
        if (retries >= 5) { Debug.LogError($"Reliable message dropped after 5 retries"); continue; }
        var recipients = ResolveRecipients(msg);
        if (recipients.Count == 0)
            _pendingReliable.Enqueue((msg, Time.time + 1f, retries + 1));
        else
            foreach (var r in recipients) r.ReceiveMessage(msg);
    }
}

2.6 状态机完整性

重要 状态机缺少恢复路径 ▶

问题描述

PlanningModule 的状态机有 10 个状态（Idle → Parsing → Grouping → ... → Active → Done/Failed），但 Failed 状态没有恢复路径。一旦任何阶段失败，模块卡死在 Failed，需要外部手动重置。

ADM 的状态也类似：Failed 后 IsIdle() 返回 true，但内部 ctx 可能处于不一致状态。

改善方案

// PlanningModule — 添加重置方法
public void Reset()
{
    if (activeCoroutine != null) { StopAllCoroutines(); }
    state = PlanningState.Idle;
    busy = false;
    parsed = null;
    myGroup = null;
    confirmedSlot = null;
    startExecReceived = false;
    agentPlan = null;
    _constraintDict.Clear();
    selections.Clear();
    selectionTs.Clear();
    occupiedSlots.Clear();
    Debug.Log($"[PlanningModule] {props?.AgentID} 已重置");
}

// 在 Failed 状态添加自动重试（可选）
private IEnumerator AutoRecovery()
{
    yield return new WaitForSeconds(5f);
    if (state == PlanningState.Failed)
    {
        Debug.Log("[PlanningModule] 尝试自动恢复...");
        Reset();
    }
}

2.7 执行层抽象

改进 AgentMotionExecutor动作完成判定 ▶

问题描述

AgentMotionExecutor 中各动作协程（DoMoveTo/DoPatrol/DoObserve等）的完成判定逻辑分散在各个方法内，且存在以下问题：

  • MoveTo 到达判定距离是硬编码的（通常 1-2m），对不同场景不灵活
  • 动作超时无全局保护，可能无限循环
  • 动作失败（如路径不可达）反馈给 ADM 的机制不完善

改善方案

// 新增: 动作执行结果
public enum ActionResult { Success, Failed, Timeout }

// 统一动作协程包装
private IEnumerator ExecuteAction(AtomicAction action)
{
    float timeout = action.duration > 0 ? action.duration : 60f;
    float startTime = Time.time;

    var innerRoutine = action.type switch
    {
        AtomicActionType.MoveTo => DoMoveTo(action),
        AtomicActionType.PatrolAround => DoPatrol(action),
        // ... others
        _ => DoUnknown(action)
    };

    while (innerRoutine.MoveNext())
    {
        if (Time.time - startTime > timeout)
        {
            Debug.LogWarning($"[AME] Action {action.actionId} timeout");
            OnActionComplete(action, ActionResult.Timeout);
            yield break;
        }
        yield return innerRoutine.Current;
    }

    OnActionComplete(action, ActionResult.Success);
}

private void OnActionComplete(AtomicAction action, ActionResult result)
{
    actionRunning = false;
    if (result != ActionResult.Success)
        adm?.OnActionFailed(action, result);
    else
        adm?.CompleteCurrentAction();
}

Part 3: 改善路线图

总体原则

先保证最小可运行闭环（单组合作场景跑通），再扩展到竞争/对抗，最后优化性能和学术包装。

Phase 1: 跑通（1-2周）

目标：单组 2-3 Agent 合作任务端到端跑通

Phase 2: 稳定（2-3周）

任务优先级涉及文件工作量

目标：竞争/对抗场景可用，论文素材收集
