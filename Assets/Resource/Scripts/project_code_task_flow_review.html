UnityLLM 脚本目录代码审查与任务流程链总览

这份文档覆盖当前可访问目录 Assets/Resource/Scripts 下的运行时脚本、前端页面、配置和设计文档， 目标不是简单罗列文件，而是把“任务从哪里进入、在谁手里变成计划、如何变成动作、如何执行、如何被监控”讲成一条完整链路， 同时指出当前实现里最值得优先修的结构性问题。

审查范围：Scripts 目录内 51 个源代码 / 文档 / 配置文件 主运行链：Campus 地图 + Planning LLM#1~#4 + ADM LLM-A/B + 物理执行 + Dashboard 未纳入逐行解释：.meta / onnx / pt / tfevents 等二进制与 Unity 资源元文件
范围与结论 关键发现 主任务链 示例走查 状态与消息 运行时代码全表 文档与配置全表 设计图与现状差异

范围与结论

当前真实主链

当前真正形成闭环的主链是： AgentSelectionManager / dashboard -> PlanningModule -> CommunicationModule/Manager -> ActionDecisionModule -> AgentMotionExecutor -> PerceptionModule -> AgentStateServer + web/dashboard.html。

地图主底座是 CampusJsonMapLoader + CampusGrid2D，旧的 MapGenerator 和多份旧流程图更像保留分支或历史方案。

整体判断

项目已经具备“多智能体任务提交、组内协商、步骤拆解、原子动作执行、局部冲突协商、Web 可视化”的完整骨架， 但仍存在三类明显断层：

  • 一类是 编译级断链，旧字段引用没有清干净。
  • 一类是 行为级断链，感知事件和记忆反思链没有真正闭环。
  • 一类是 文档级漂移，部分 Mermaid 图和旧说明仍在讲已被替换的类名和流程。

关键发现

P0 编译断链

LLM_Modules/ActionDecisionModule.cs:88 和 Other_Modules/AgentStateServer.cs:204 仍在访问 PlanningModule.currentMission，但当前 LLM_Modules/PlanningModule.cs 已没有这个字段定义。

这不是“逻辑可能不准”，而是静态搜索层面就已经能确认的旧接口残留。只要 Unity 重新编译到这些文件，当前代码就会在这一层卡住。

P0 密钥泄露

LLM_API/LLMInterface.cs:61 和 LLM_API/LLMInterface.cs:93 在默认配置里写入了真实形态的 API Key。

这意味着源码、日志、Inspector 序列化、多人协作仓库都会直接暴露密钥，必须立刻改成环境变量、外部配置或本地忽略文件注入。

P1 感知重规划失效

PerceptionModule 会把敌方和临时障碍事件送到 ActionDecisionModule.OnPerceptionEvent()， 见 Other_Modules/PerceptionModule.cs:507 和 Other_Modules/PerceptionModule.cs:598。

但 LLM_Modules/ActionDecisionModule.cs:607-610 里的 EventRequiresReplan() 永远返回 false， 所以这些事件会入队，但运行态不会因事件真正触发 LLM-B 重规划。

P1 记忆/反思链未真正接入主链

MemoryModule 和 ReflectionModule 实现得很完整， 但跨文件调用检索显示，真正被运行链使用的只有 AgentLLMControl.RecordEvent() -> MemoryModule.AddMemory() 这种旧接口。

当前没有看到 RememberPlanSnapshot、 RememberActionExecution、 NotifyBlocked、 GetActionGuidance 等被 Planning / ADM / 执行层稳定调用， 所以“学习反思闭环”更多停留在模块已实现、但未接线的状态。

P2 文档与实际实现漂移

多份设计图仍在使用 CoordinationManager、 MLAgentsController、 AnalyzeMissionDescription、 ReceiveMissionAssignment 等旧名词， 而实际代码已经切到 CommunicationManager、 AgentMotionExecutor、 RunLLM1-4 与四阶段槽位协商。

P2 UI / 工作区存在历史残留

Agents/UIController.cs 的手动控制按钮逻辑为空， .vscode/launch.json 指向的是无关的 C 程序路径， 说明仓库里仍混有历史实验或个人工作区残留配置。

主任务流程链

下面这条链是按当前代码静态串出来的“真实执行版”，每一步都标了输入、处理中枢、状态变化、输出和下游影响。

入口层：任务是怎么进入系统的
  • 本地 UI 入口：Other_Modules/AgentSelectionManager.cs 选中某个 Agent 后调用 PlanningModule.SubmitMissionRequest(desc, agentCount)。
  • Web 入口：web/dashboard.html 调 POST /api/task，后台由 AgentStateServer 入队，再在 Unity 主线程找到一个空闲 PlanningModule 提交任务。
  • 假设前提：场景里至少已有一个可工作的智能体与 CommunicationManager / LLMInterface。
规划阶段 1：自然语言任务被解析成结构化任务
  • 执行者：PlanningModule.RunLLM1()。
  • 输入：任务描述 + 智能体数量。
  • 处理：调用 LLMInterface.SendRequest()，要求 LLM 返回 ParsedMission JSON，核心字段是 relType / groupCnt / groupMsns / timeLimit。
  • 状态变化：PlanningState.Idle -> Parsing。
  • 输出：本轮任务 ID msn_xxx 与分组草案。
规划阶段 2：本地分组并广播 GroupBootstrap
  • 执行者：PlanningModule.AssignGroups()。
  • 处理：根据 groupCnt 把当前已注册的 Agent ID 分组，再通过 SelectLeader() 用电量最高者做组长。
  • 消息：经 CommunicationModule.SendScopedMessage(Public, GroupBootstrap) 广播给所有智能体。
  • 状态变化：Parsing -> Grouping。
  • 风险：组长选择依赖所有 Agent 已经注册进 CommunicationManager；仓库里已有 p2_design_issues.html 专门记录这一类时序问题。
规划阶段 3：组长生成 PlanSlot，成员选择槽位
  • 组长走 PlanningModule.RunLLM2()：基于组任务和成员数，让 LLM 生成与成员数等长的 PlanSlot[]。
  • 成员走 PlanningModule.RunLLM3()：结合自己位置、电量，从可选槽中挑一个最适合自己的槽位。
  • 消息链：SlotBroadcast -> SlotSelect -> SlotConfirm。
  • 状态变化：组长常见是 SlotGen，成员常见是 SlotPick。
  • 冲突解决策略：当前实现是“组长先处理自己，再按选择到达时间先到先得”，冲突者被分配到剩余槽位。
规划阶段 4：确认槽位后拆成步骤
  • 执行者：PlanningModule.RunLLM4()。
  • 输入：某个 Agent 已确认的 PlanSlot。
  • 处理：把槽位描述 desc 拆成顺序化的 PlanStep[]，并把协同约束绑定到最合适的步骤上。
  • 输出：AgentPlan，内部有 steps[] 与 curIdx。
  • 状态变化：StepGen -> Active。
自治触发门：什么时候从“有步骤”变成“开始决策”
  • 执行者：Agents/IntelligentAgent.cs。
  • 处理：每 2 秒检查一次 ShouldMakeDecision()。当 PlanningModule.HasActiveMission() 为真且当前状态是 Idle，就启动 MakeDecisionCoroutine()。
  • 它本质上是“任务节拍器”，不负责推理内容，只负责决定何时唤起 ADM。
执行规划：PlanStep 被 ADM 解释成 AtomicAction[]
  • 执行者：LLM_Modules/ActionDecisionModule.cs。
  • 链路：DecideNextAction() -> StartStep() -> RunLLMA()。
  • 输入：当前步骤文本、角色、当前位置、协同约束、周边相对地图、当前感知快照。
  • 地图文本来源：MapTopologySerializer.GetAgentRelativeMap()。
  • 输出：AtomicAction[]，动作类型包括 MoveTo / PatrolAround / Observe / Wait / FormationHold / Broadcast / Evade。
  • 状态变化：ADMStatus.Idle -> Interpreting -> Negotiating。
协商窗口：先广播自己的动作目标，再决定是否需要重规划
  • 执行者：ActionDecisionModule.BroadcastContextUpdate()。
  • 处理：把自己当前动作目标集合整理成 AgentContextUpdate，通过 MessageType.BoardUpdate 发给同队。
  • 黑板缓存：每个 Agent 在本地维护 blackboard，记录队友当前位置、角色、当前动作、计划目标。
  • 协商结果：CheckConflictsAndMaybeReplan() 统计目标冲突，按“总冲突数更大优先、再按 AgentID 字典序”选出 resolver，必要时走 RunLLMB() 重规划。
物理执行：AtomicAction 被刚体控制器逐条消费
  • 执行者：Other_Modules/AgentMotionExecutor.cs。
  • 轮询方式：每帧从 adm.GetCurrentAction() 取当前动作，动作变了就启动对应协程。
  • MoveTo 会把地名解析到 CampusGrid2D 的格子，再调用 A*，最终转成世界坐标 waypoint。
  • Observe 会强制调用 PerceptionModule.SenseOnce()。
  • 动作结束后回调 adm.CompleteCurrentAction()，当动作队列走完，ADM 再通知 PlanningModule.CompleteCurrentStep()。
感知与反馈：环境变化怎样反向影响执行
  • 执行者：Other_Modules/PerceptionModule.cs。
  • 输入：基于 Agent 类型的射线扫描，检测层按障碍 / 资源 / Agent 分层处理。
  • 输出一：更新 AgentDynamicState.NearbyAgents 和 DetectedSmallNodes。
  • 输出二：登记到全局 SmallNodeRegistry，供后续查询。
  • 输出三：敌方或临时障碍会向 ADM 发事件，但当前默认实现不会因这些事件真正触发重规划，这是上面列出的 P1 问题。
监控与运维：系统如何把运行态暴露给浏览器
  • 执行者：Other_Modules/AgentStateServer.cs。
  • 主线程定期采样 Agent 快照、消息日志、位置历史、地图元数据、LLM 日志。
  • 后台线程用 HttpListener 提供 /api/state、/api/messages、/api/history、/api/map、/api/gridmap、/api/llm-logs、/api/task、/api/model。
  • 前端页面 web/dashboard.html 把这些数据画成 2D 地图、Agent 卡片、通信面板、LLM 日志和控制台。

示例走查

下面用一个当前代码最容易映射的例子，把“任务链”翻译成你能跟着代码走的故事。

例子：4 架无人机执行“侦察 A 楼北侧并实时共享异常”

1. 用户在 AgentSelectionManager 或 dashboard 输入任务： “4 架无人机侦察 A 楼北侧并实时共享异常” 2. PlanningModule.RunLLM1() 可能得到： { "relType": "Cooperation", "groupCnt": 1, "groupMsns": ["协作侦察A楼北侧并共享异常信息"], "timeLimit": 300 } 3. 组长 RunLLM2() 可能得到 4 个槽位： - s0 Scout: 从东路接近 A 楼北侧 - s1 Scout: 从西路接近 A 楼北侧 - s2 Supporter: 在高空维持中继 - s3 Defender: 在北门附近待命并回传异常 4. 每个成员 RunLLM3() 选槽。 比如离东路更近、电量更高的 drone_01 选择 s0。 5. drone_01 的 RunLLM4() 可能得到步骤： - step_1: 从当前位置飞往 A 楼北侧 - step_2: 对 A 楼北侧进行侦察 - step_3: 广播侦察结果 6. ActionDecisionModule.RunLLMA() 继续把 step_1 变成动作： - aa_1 Evade: 上升到安全高度 - aa_2 MoveTo: 飞往 A楼北侧 - aa_3 Observe: 观察 15 秒 - aa_4 Broadcast: 向队友广播侦察状态 7. AgentMotionExecutor 执行： - MoveTo -> CampusGrid2D 解析地名 + A* - Observe -> 触发 PerceptionModule.SenseOnce() - Broadcast -> BoardUpdate 广播给队友 8. 如果队友和自己都计划去同一目标， ADM 会在 Negotiating 阶段用黑板比较 plannedTargets， 必要时由 resolver 调用 RunLLMB() 重新生成动作队列。

这个例子里地图如何参与

  • CampusJsonMapLoader 把校园 JSON 建成可视地物。
  • CampusGrid2D 把这些地物离散成可寻路网格，并保留地名、uid、空间画像。
  • MapTopologySerializer 把地图压成 LLM 能读的相对文本，而不是把整个网格丢给模型。

这个例子里通信如何参与

  • 规划期消息：GroupBootstrap / SlotBroadcast / SlotSelect / SlotConfirm / StartExecution。
  • 执行期消息：BoardUpdate。
  • 监控期日志：CommunicationManager.RecentMessageLog 被仪表板读取。

状态与消息

PlanningState

Idle -> Parsing -> Grouping -> SlotGen / SlotPick -> StepGen -> Active -> Done / Failed

其中 PlanningModule 真正对外暴露给 ADM 的只有三个关键接口： HasActiveMission()、 GetCurrentStep()、 CompleteCurrentStep()。

ADMStatus

Idle -> Interpreting -> Negotiating -> Running -> Interrupted / Replanning -> Done / Failed

这层是“步骤内执行状态”，不是任务总状态。一个任务完成会经过多个 ADM 周期。

主消息类型

  • GroupBootstrap：全局分组通知。
  • SlotBroadcast：组长发可选槽位。
  • SlotSelect：成员回报选择。
  • SlotConfirm：组长确认最终槽位。
  • StartExecution：全组开始步骤拆解。
  • BoardUpdate：执行态黑板更新。

当前链路的边界保护

  • PlanningModule.CheckTimeout() 防止长时间卡死在中间状态。
  • ActionDecisionModule.MaxReplanCount 防止黑板冲突无限重规划。
  • CommunicationManager 对消息日志做 2000 条上限裁剪。
  • SmallNodeRegistry 对动态节点做 TTL 清理。

设计图与现状差异

现在代码里真正活着的术语

  • CommunicationManager，不是 CoordinationManager。
  • AgentMotionExecutor，不是 MLAgentsController。
  • RunLLM1 / RunLLM2 / RunLLM3 / RunLLM4，不是 AnalyzeMissionDescription / ParseAndCreatePlan。
  • GroupBootstrap / SlotBroadcast / SlotSelect / SlotConfirm / StartExecution，不是单次“任务广播 -> 接收任务”老链。

还停留在“设计意图”的能力

  • 记忆驱动 Planning prompt。
  • 反思驱动 ADM 策略调整。
  • 感知事件驱动真实重规划。
  • 旧 ML-Agents 与当前 LLM 主链的统一桥接。

一句话结论

如果你要向别人解释这个项目，最稳妥的说法不是“它已经完整实现了学习反思闭环”， 而是：“它已经完整实现了 LLM 任务协商 + 原子动作执行 + Web 监控 的主框架， 并且已经预留了 记忆、反思、感知重规划 的模块，但后两者还没有完全接上线。”
