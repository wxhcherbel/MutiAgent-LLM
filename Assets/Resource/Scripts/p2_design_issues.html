BUG-07
竞争/对抗模式的分组机制缺乏用户意图表达 Medium
PlanningModule.cs · AssignGroups()
代码位置
PlanningModule.cs AssignGroups() RunLLM1()
问题是什么

当用户提交"A队攻击B队"这类对抗任务时，LLM#1 会把 relType 设为 Adversarial、groupCnt 设为 2。但 AssignGroups 分组时，直接从 CommunicationManager.GetAllAgentIds() 拿到所有无人机 ID，然后按注册顺序均分成两队——完全不考虑"哪几架是红方、哪几架是蓝方"这个用户意图。

// PlanningModule.cs — AssignGroups() 核心逻辑
string[] agentIds = CommunicationManager.Instance
    .GetAllAgentIds(); // ["drone_1","drone_2","drone_3","drone_4","drone_5","drone_6"]

// ← groupCnt=2，baseSize=3，直接按 index 切
int baseSize = n / groupCnt;   // 6/2 = 3
for (int g = 0; g < groupCnt; g++)
{
    int size = baseSize + (g == 0 ? remainder : 0);
    string[] members = agentIds.Skip(idx).Take(size).ToArray();
    // 组0: ["drone_1","drone_2","drone_3"]
    // 组1: ["drone_4","drone_5","drone_6"]
    // ← 如果用户意图是 drone_1/3/5 为红方，完全无法实现
}
复现场景：6 架无人机对抗

用户在 Unity 场景中想让"偶数编号机"攻、"奇数编号机"守。但注册顺序是 drone_1 到 drone_6，系统将 1/2/3 分为一队、4/5/6 分为另一队，完全不符合用户意图。更严重的是，分组后 TeamID 被覆盖（props.TeamID = groupIndex），原有的 AgentProperties 中设好的队伍信息丢失。

建议修复方向
  • UI 层提供"分队选择"：用户拖拽智能体到不同队伍槽位，最终以 List<string[]> teamGroups（每组的 ID 列表）传给 SubmitMissionRequest。
  • 或者：合作任务继续用全体 ID 均分；竞争/对抗任务强制要求用户在 UI 里指定分队。
  • 最小改动：在 GroupBootstrapPayload 里附带 relType，接收方根据自己原始 TeamID 过滤，不覆盖 TeamID。
BUG-08
黑板条目永不过期，已完成的队友制造幽灵冲突 Medium
ActionDecisionModule.cs · blackboard
代码位置
ActionDecisionModule.cs blackboard Dictionary CheckConflictsAndMaybeReplan()
问题是什么

黑板（Dictionary<string, AgentContextUpdate> blackboard）里的条目只会被新广播覆盖，但永远不会被删除。当某架无人机完成任务（进入 Done 状态）后，它不再广播新的 BoardUpdate，但它的旧条目仍然留在其他所有机器的黑板里，继续参与冲突检测和选举。

t=0 drone_A 广播 targets=[教学楼A, 实验室] → 写入所有人的 blackboard t=5 drone_A 完成任务 → ADMStatus.Done → 不再广播 t=10 drone_B 的 LLM-A 规划出 targets=[教学楼A] drone_B 扫描黑板：drone_A 的旧条目仍含 "教学楼A" → 误判为冲突 → 触发不必要的 resolver 选举 → RunLLMB drone_A 实际上早已离开，什么冲突都没有！
频率估计

合作任务中，越靠后完成的无人机越容易被已完成机器的"幽灵条目"干扰。任务越长（步骤越多），积累的幽灵冲突越严重，可能导致后期无人机反复触发 RunLLMB 而永远无法达到 Running 状态。

建议修复方向
  • 方案 A（推荐）：时效过滤。 在冲突扫描时，跳过 Time.time - kv.Value.timestamp > 30f 的条目。
  • 方案 B：主动清空广播。 在 CompleteCurrentAction() 进入 Done 状态时，广播一条 plannedTargets = [] 的 BoardUpdate，通知其他机器清空本机占位。
// 方案 A：在扫描时忽略过期条目（30 秒视为失效）
foreach (var kv in blackboard)
{
    if (Time.time - kv.Value.timestamp > 30f) continue; // 过期，跳过
    if (string.Equals(kv.Key, myId, ...)) continue;
    myConflict += CountConflicts(kv.Value.plannedTargets, ownTargets);
}
BUG-09
SelectLeader 依赖不稳定的实时电量读取路径 Medium
PlanningModule.cs · SelectLeader()
代码位置
PlanningModule.cs SelectLeader()
问题是什么

组长选举通过读取各无人机的实时电量来决定谁最有资格。读取路径如下：

// 调用链：4 层嵌套，每层都可能返回 null
CommunicationManager.Instance          // ① 单例是否初始化？
  .GetAgentModule(id)                  // ② 该 ID 是否已注册？
  .GetComponent<IntelligentAgent>()   // ③ 组件是否挂载？
  .CurrentState                        // ④ CurrentState 是否初始化？
  .BatteryLevel
问题场景：Unity Start() 执行顺序不确定

协调者调用 AssignGroups → SelectLeader 时，某些无人机的 CommunicationModule.Start() 可能还没有执行（Unity 不保证多 GameObject 的 Start 顺序），导致 GetAgentModule(id) 返回 null。此时所有 agent 的电量都是 0，组长默认回退为 memberIds[0]（注册顺序第一个），而非真正电量最高的 agent。

更隐蔽的问题：每次 reltype 相同但结果不同

如果相邻两次任务提交的时间点不同（第一次在场景刚加载时，第二次在稳定运行后），因为 Start() 初始化顺序不同，SelectLeader 每次可能返回不同的 agent 作为组长，导致任务行为不可重现。

建议修复方向
  • 加 null 检查后使用 ID 字典序作为 fallback（确定性更强）：
private string SelectLeader(string[] memberIds)
{
    string leaderId  = memberIds[0];
    float  maxBattery = float.MinValue;

    foreach (string id in memberIds)
    {
        var mod = CommunicationManager.Instance?.GetAgentModule(id);
        float bat = mod?.GetComponent<IntelligentAgent>()?.CurrentState?.BatteryLevel ?? -1f;

        // bat == -1 表示读取失败，回退到字典序比较确保确定性
        if (bat > maxBattery ||
            (bat == -1f && maxBattery == -1f &&
             string.Compare(id, leaderId, StringComparison.Ordinal) > 0))
        {
            maxBattery = bat;
            leaderId   = id;
        }
    }
    return leaderId;
}
BUG-10
非协调者的 parsed 只有 msnId，其余字段为空 Medium
PlanningModule.cs · OnGroupBootstrap()
代码位置
PlanningModule.cs OnGroupBootstrap()
问题是什么

当非协调者（普通成员）收到 GroupBootstrap 广播时，代码只给 parsed 创建了一个只含 msnId 的空对象，其他字段（relType, timeLimit, groupCnt）全是 null/default：

// OnGroupBootstrap() — 非协调者分支
if (parsed == null)
    parsed = new ParsedMission { msnId = p.msnId }; // ← relType=null, timeLimit=0
else if (p.msnId != parsed.msnId)
    return;
// parsed.relType 仍为 null！后续若有逻辑读此字段会出错
当前影响

现有代码中，非协调者后续只用 parsed.msnId（用于消息匹配），所以目前不会崩溃。但如果未来扩展功能（比如非协调者根据 relType 决定是否与敌方通信），parsed.relType == null 会引发 NullReferenceException 或逻辑错误。这是一个潜伏的时间炸弹。

建议修复
// 将 GroupBootstrapPayload 中的字段同步到非协调者的 parsed
if (parsed == null)
{
    parsed = new ParsedMission
    {
        msnId     = p.msnId,
        relType   = p.relType,   // ← 同步进来
        groupCnt  = p.groups?.Length ?? 1
    };
}
else if (p.msnId != parsed.msnId)
    return;
BUG-11
JsonUtility（发送）与 Newtonsoft.Json（接收）混用，嵌套结构有序列化风险 Medium
CommunicationModule.cs · SendScopedMessage()
代码位置
CommunicationModule.cs SendScopedMessage() TryParseMessageContent()
问题是什么

整个项目存在两套 JSON 序列化器并行使用的情况：

消息发送路径（CommunicationModule）： payload → JsonUtility.ToJson() → AgentMessage.Content(string) 消息接收路径（CommunicationModule）： AgentMessage.Content → JsonUtility.FromJson<T>() → payload ✓（匹配） LLM 输出解析（PlanningModule / ADM）： llmResult(string) → JsonConvert.DeserializeObject<T>() → actions ✓（不冲突）
为什么说有"风险"而不是直接崩溃

对于简单 [Serializable] 类，JsonUtility 与 Newtonsoft 输出格式基本一致，多数情况正常工作。但以下情形下 JsonUtility 会悄悄丢数据或崩溃：

  • 多层嵌套数组： GroupBootstrapPayload → GroupDef[] → string[] memberIds，Unity 2019 及更早版本的 JsonUtility 对此支持有 bug（2020+ 已修复）。
  • null 字段： JsonUtility 不序列化 null，反序列化时该字段保持 default 值（如空字符串），而 Newtonsoft 会正确处理 null。若接收方用 Newtonsoft 反序列化 JsonUtility 输出，null 字段行为不一致。
  • Dictionary 字段： JsonUtility 完全不支持 Dictionary，如果未来 payload 类里加了 Dictionary 字段，会悄悄丢弃。
建议修复

将 CommunicationModule.SendScopedMessage 和 TryParseMessageContent 统一改用 Newtonsoft.Json，与 PlanningModule 保持一致：

// 发送（SendScopedMessage 内）
Content = payload != null
    ? JsonConvert.SerializeObject(payload)   // ← 改这里
    : "{}",

// 接收（TryParseMessageContent 内）
payload = JsonConvert.DeserializeObject<TPayload>(message.Content);  // ← 改这里

注意：Newtonsoft.Json 已在项目中引入（PlanningModule.cs 顶部 using Newtonsoft.Json），直接引用即可，无需额外安装。

BUG-12
WaitSec=25f 全局超时可能不足以覆盖完整 LLM 调用链 Medium
PlanningModule.cs · CheckTimeout()
代码位置
PlanningModule.cs CheckTimeout() WaitSec = 25f
问题是什么

WaitSec = 25f 是一个所有中间阶段共用的超时时间。只要从进入当前状态开始超过 25 秒，就会被判 Failed。但完整的 4 阶段协商涉及多次 LLM 请求：

完整流程时间估算（每次 LLM 5-8 秒）： 阶段 状态 等待内容 耗时估算 ───────────────────────────────────────────────────────────── LLM#1 Parsing 解析任务文本 ~5-8s Grouping 本地分组（同步） <1s LLM#2 SlotGen 生成计划槽（组长） ~5-8s LLM#3 SlotPick 全体成员选槽（并发） ~5-8s (组长等待) 等所有成员 SlotSelect 到达 +网络延迟 LLM#4 StepGen 步骤拆解 ~5-8s ───────────────────────────────────────────────────────────── 最坏情况总计：4×8s + 等待 = ~35-40s > WaitSec=25s
超时会发生在哪个阶段

CheckTimeout() 每帧调用，检测从 SetState()（会更新 waitStart）到现在的时间。如果 LLM#3 阶段（SlotPick）有 6 台无人机，最慢的一台等待到第 24 秒才发 SlotSelect，而组长还没收齐所有选择，此时已超时，整个任务被判 Failed——即使 LLM 其实没有问题，只是网络稍慢。

另一个问题：SetState 会重置 waitStart

每次 SetState(s) 都会执行 waitStart = Time.time。这意味着超时是每个阶段独立计时的，单阶段超时才触发 Failed，而不是整个流程总时长。这实际上是合理设计，但 25s 对于 SlotPick 阶段（等待多个成员的网络回包）来说仍然可能偏短。

建议修复
  • 将 WaitSec 提升到 60f 或 120f（或者改为可配置的 Inspector 字段）。
  • 或者：针对不同阶段设置不同超时：等待 LLM 的阶段（Parsing/SlotGen/StepGen）超时 60s，等待消息的阶段（SlotPick）超时 30s × memberCount。
// 最小改动：按阶段动态计算超时
private float GetTimeoutForState(PlanningState s) => s switch
{
    PlanningState.Parsing  => 60f,
    PlanningState.SlotGen  => 60f,
    PlanningState.SlotPick => 30f * (myGroup?.memberIds?.Length ?? 1),
    PlanningState.StepGen  => 60f,
    _                      => 30f
};
