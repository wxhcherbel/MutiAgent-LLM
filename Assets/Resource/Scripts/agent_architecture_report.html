摘要

本报告对现有基于 Unity3D 的多智能体协同仿真系统进行全面架构审查，对标 Park et al. (2023) 提出的"生成式智能体"（Generative Agents，即"斯坦福小镇"）研究范式，系统分析实现自主行为与涌现所需的模块、机制和架构改造方案。当前系统已具备完善的约束驱动多智能体协作、LLM 规划流水线与 MAD 辩论机制，但在自主目标生成、记忆检索评分、层级反思机制、社会关系建模等方面存在显著差距。

⚠ 核心洞察（本次报告更新重点）：无人机协同系统与斯坦福小镇存在根本差异——前者需要物理协作（时序严格、方向互斥、角色分工），后者仅需社交涌现。因此，当前系统的分组机制与Slot系统是正确且必要的，不应追求纯粹独立涌现。正确的演进方向是：保留角色框架层（LLM#1/2/3），去掉固定步骤预生成层（LLM#4），将执行层（LLM#5/ADM）升级为响应式感知-决策循环，并引入记忆/反思/社会关系等上下文模块为其提供「燃料」。

本报告围绕以下核心问题展开：（1）分组和Slot选择对于无人机协作的必要性分析；（2）记忆模块与反思模块如何升级为符合研究范式的完整实现；（3）除现有模块外，还需新增哪些关键模块及其集成方式；（4）通过端到端实战案例说明每个模块的实际贡献。研究引用了 Generative Agents、CoALA、VOYAGER、ReAct、MemoryBank 等 12 篇代表性论文，并结合当前代码库的具体实现细节提出可落地的改造方案。

1.4 无人机协作的涌现悖论：Slot系统不是约束，而是协作基础

核心疑问解析：「如果想实现斯坦福小镇式涌现，分组和选Slot有没有用？每个智能体都各自涌现的话，他们怎么协作，不就变成独立个体了？」—— 这个问题触及了协作型多智能体系统的核心矛盾。

🏘️ 斯坦福小镇：为什么可以独立涌现？

斯坦福小镇的智能体天然不需要物理协作：

  • Isabella「想办情人节派对」— 不需要Bob同一时刻在特定位置做特定动作
  • 社会协调通过对话自然发生，时间窗口极其宽松
  • 错过某次互动代价很低，可以明天再谈
  • 根本没有物理同步需求，因此独立涌现不会导致任务失败

🚁 无人机：为什么必须物理协调？

无人机协作任务有硬性物理同步约束：

  • 三机包围目标 — 必须同时在三个方向，缺一不可
  • 拦截任务 — 侦察机确认后拦截机才能接近，时序严格
  • 若全员自由涌现去「侦察」→ 没有人拦截，任务直接失败
  • A和B同方向接近 → 目标逃脱，且南侧无人封锁

✅ 正确理解：「涌现」发生在哪一层？

涌现可以在不同层级发生。无人机系统选择在执行层涌现，而不是角色层：

层1：角色层（结构化，必要）

Slot系统定义「这次任务需要 Scout + Interceptor + Perimeter Guard」。这是协作的框架，不能涌现——否则全员做同一件事，协作崩溃。

保留 LLM#2
层2：角色认领（轻度涌现）

每架无人机基于自身当前状态（位置、电量、技能历史）自主选择最适合自己的Slot。UAV-1电量满且在北门附近 → 自然选 Scout。

优化 LLM#3
层3：执行层（完全涌现）

Scout 从哪个高度观察、何时切换到悬停、遇到障碍怎么绕行——完全由 ADM 响应式决策，不预先规定步骤。这与斯坦福小镇的涌现机制完全相同。

替换 LLM#4 → 升级 LLM#5
足球队需要前锋、中场、后卫（角色层，结构化）。球员争抢哪个位置（角色认领，轻度涌现）。球员在场上如何跑位传球（执行，完全涌现）。没有角色分工的球队不叫「涌现」，叫「混乱」。

✅ 当前系统中值得保留的部分

  • 分组（Grouping）：正确。定义团队边界，竞争/合作场景中必要。
  • Slot 生成（LLM#2）：正确。描述任务的角色需求，是协作不退化为独立个体的核心。
  • Slot 认领（LLM#3 + Whiteboard）：基本正确。可进一步优化为基于实时状态的自主竞标。
  • C2/C3 约束系统：正确。物理同步的保障——等价于斯坦福小镇中的时空隐式约束。
  • SharedWhiteboard：正确。共享感知基础设施，类似斯坦福小镇的「共享环境状态」。

⚡ 需要改进以实现「执行层涌现」

  • LLM#4（步骤预生成）：最大改进点。提前生成固定Steps，环境变化时计划立即过时。建议去掉，改为响应式ADM。
  • LLM#3（槽位选择）：可优化。由集中式LLM决策改为基于实时状态的自主选择（白板先到先得）。
  • 感知→记忆链路：待完善。感知结果目前不自动写入记忆流，影响ADM实时决策质量。
改进核心：去掉 LLM#4，升级 LLM#5（ADM）为主决策引擎。输入从「固定步骤序号」改为「Slot角色描述 + 实时感知 + 白板状态 + 记忆上下文」，输出为下一个原子动作。

2.4 涌现行为案例分析

Valentine's Day 案例

研究者仅向一名智能体注入一条倾向信息："想办一个情人节派对"。随后系统中自发涌现出：

  1. 信息扩散：该智能体在两天内主动邀请其他智能体参加派对，形成口耳相传链
  2. 社会协调：智能体之间自发互相认识，询问对方是否参加
  3. 情感响应：部分智能体在日记中记录对派对的期待感
  4. 自主安排：智能体提前调整日程，预留时间参加派对
  5. 社会关系演化：多对智能体形成新的"约会"关系，超出原始设计预期
以上所有行为均无任何显式编程，完全由记忆流、反思和规划的交互涌现而来。

5.1 三维检索评分公式

Generative Agents 检索评分公式（Park et al., 2023）
score(m, q) = α₁ · recency(m) + α₂ · importance(m) + α₃ · relevance(m, q)
其中各分量归一化至 [0, 1]，α₁ = α₂ = α₃ = 1.0（原论文设定）

recency(m) — 时效性

recency(m) = e−λ·Δt
λ = 0.995（每小时衰减）
Δt = 距上次访问的小时数

最近的记忆权重高，越久远的记忆越难被检索到。指数衰减模拟人类遗忘曲线特征。

float Recency(Memory m) {
  float hours = (Time.time - m.lastAccessedAt) / 3600f;
  return Mathf.Exp(-0.995f * hours);
}

importance(m) — 重要性

importance(m) = LLM_score(m) / 10.0
LLM 评分 1–10，归一化至 [0,1]
1=日常琐事，10=极为重要

每次创建记忆时，异步调用 LLM 对其重要性打分并缓存。重要的事件（紧急事故、关键决策）得高分。

// LLM Prompt:
"On a scale of 1 to 10, how
 important is this memory?
 1=mundane, 10=critical.
 Memory: {summary}
 Output: {\"score\": N}"

relevance(m, q) — 相关性

relevance = cos(embed(m), embed(q))
余弦相似度
embed() = 文本向量化函数

将记忆描述和查询问题都转为嵌入向量，计算余弦相似度。在不支持云端嵌入时可用 TF-IDF 近似。

float Cosine(float[] a, float[] b) {
  float dot = 0, na = 0, nb = 0;
  for (int i=0; i<a.Length; i++) {
    dot += a[i]*b[i];
    na += a[i]*a[i]; nb += b[i]*b[i];
  }
  return dot / (Sqrt(na)*Sqrt(nb));
}

5.4 艾宾浩斯遗忘曲线集成（来自 MemoryBank）

参照 Zhong et al. (2023) MemoryBank 的设计，为每条记忆维护记忆强度（memory strength），模拟人类记忆的遗忘与巩固：

记忆强度更新公式（基于艾宾浩斯遗忘曲线）
strength(t) = e−t / (stability · e0.1·difficulty)
t = 距上次记忆的时间；stability = 基础稳定性；difficulty = 内容复杂度
每次访问/检索：stability += Δstability（记忆被强化）
// 在 MemoryModule.cs 中新增
public void AccessMemory(string memoryId) {
  var m = _memories.Find(x => x.id == memoryId);
  m.lastAccessedAt = Time.time;
  m.stabilityScore += 0.1f;  // 访问增强记忆稳定性
  m.strengthScore = ComputeStrength(m);
}

float ComputeStrength(MemoryEntry m) {
  float t = (Time.time - m.lastAccessedAt) / 3600f;
  return Mathf.Exp(-t / (m.stabilityScore * Mathf.Exp(0.1f * m.difficulty)));
}

  1. A 与 B 多次共同执行巡逻任务（记忆中积累共同经历）
  2. 反思：A 认识到"与B协作时效率更高"，B 有类似洞察

  4. 日计划生成时，两者越来越倾向于同时段规划同区域活动
  5. 搭档关系自然涌现，无需任务分配约束

涌现路径三：紧急事件级联响应

  1. 人类注入：紧急事件"北门出现未知入侵者"
  2. 感知到事件的 Agent_A 提升此记忆重要性，触发反思
  3. A 的日计划重新排序，优先处理安全任务
  4. A 通过社会关系主动联络高信任度同伴
  5. 多个智能体自发形成响应协作，无需中央调度

涌现路径四：规范形成（Norm Emergence）

  1. 某个时段多个智能体频繁使用同一充电站
  2. 经历冲突后，通过 MAD 辩论形成非正式优先级规则
  3. 各智能体将规则写入程序记忆（SkillLibrary）
  4. 后续智能体观察到这种行为模式，通过反思习得规范
  5. 群体规范自发形成，无需预先定义

核心结论：斯坦福小镇的涌现靠「每个个体自由行动然后碰撞产生社会现象」。无人机系统的涌现靠「角色框架约束整体，执行层自由感知-决策，新模块提供上下文燃料（记忆/关系/技能）」。两者都是涌现，但机制不同，都是正确的选择。

十一、RALLY 架构借鉴：预决策意图共享 + 自然语言通信改造方案

RALLY（Role-Adaptive LLM-Driven Yoked Navigation，arXiv:2507.01378）是目前与本系统最接近的公开 UAV+LLM 多智能体框架。需要先说明一个事实：本系统已经具备 RALLY 所谓的"三层架构"——ADM rolling loop 输出语义动作名（如 MoveTo("北门")），AME 自主执行 A*/APF/PD 控制，Rigidbody 处理底层物理，三层已然存在。RALLY 对本系统真正有增量价值的是另外两点：① 预决策意图共享（两阶段共识），让智能体在行动前先交换意图、再修订决策；② 自然语言通信替代枚举消息，简化 56+ 消息类型的维护负担。

11.1 预决策意图共享（两阶段共识，RALLY 核心创新）

当前系统的协调方式：事后约束

当前系统通过白板 + C1/C2/C3 约束实现协调，本质是"行动后检查"：

智能体独立决策
→
写占位到白板
→
其他智能体读白板
→
检测冲突→Wait/Replan

问题：每个智能体的 LLM 在决策时只知道队友"已经做了什么"（历史占位），不知道队友"接下来打算做什么"（意图）。这导致冲突在行动层面才被发现，需要额外的等待或重规划来消解。

RALLY 的方案：预决策意图共享（两阶段）

RALLY 将每次决策拆分为两阶段，核心在于：先形成初步意图并广播，再综合邻居意图修订最终决策。

阶段 1 — LLMinit（初步意图生成）

每个智能体基于本地观测，由 LLM 生成初步意图（"我打算去北门侦察"），立即向邻居广播。此阶段各自独立，不等待他人。

阶段 2 — LLMcons（共识修订）

收集邻居的初步意图后，再次调用 LLM，将队友意图纳入上下文，修订自己的最终决策。冲突在执行前就被感知并规避。

本地独立决策
→
广播初步意图
→
LLMcons
综合邻居修订
→
执行最终动作

与当前系统的本质差异

维度当前系统RALLY 两阶段
协调时机 行动之后：写白板占位 → 后来者检测冲突 行动之前：先广播意图 → 综合他人意图再决策
LLM 上下文 队友的历史动作（已完成的事） 队友的当前意图（即将做的事）
冲突发现时机 执行层（Wait/Replan 消解） 决策层（直接规避，不产生冲突）
LLM 调用次数 1次/batch（rolling loop） 2次/batch（init + consensus），但冲突减少可降低 Replan 频率

如何在当前系统中实现

最小化改造路径：将 rolling loop 的单次 LLM 调用拆分为两步，中间插入意图广播和收集。

ActionDecisionModule.cs（核心改造）
  • Step A：调用 LLM（简化 prompt，只含本地状态），输出初步意图 JSON：{"intent": "前往北门侦察", "targetName": "北门"}
  • Step B：通过 SharedWhiteboard 广播此意图，等待 intentBroadcastWindowMs（如 500ms）收集邻居意图
  • Step C：将收集到的邻居意图追加到完整 prompt，再次调用 LLM，输出最终 nextActions[]
  • 若无邻居意图（单独执行），Step A 结果直接作为 Step C 输出，跳过第二次 LLM 调用
SharedWhiteboard.cs（新增意图槽位）
  • 新增 Dictionary<string, IntentEntry> currentIntents：key = agentId，value = 意图文本 + 时间戳
  • IntentEntry：{ intent, targetName, timestamp }，TTL = 本 rolling 迭代周期
  • 提供 PublishIntent(agentId, intent) 和 GetNeighborIntents(agentId) 接口
  • 意图仅在本 batch 有效，下一轮自动清除（避免过时意图干扰决策）
与现有白板的区别：当前白板存储的是"已完成动作的占位声明"（C3互斥、DoneSignal等），意图槽位存储的是"本轮即将执行的初步计划"。两者并存，语义不同，不应合并。
预期效果：减少因意图冲突导致的 Wait/Replan 轮次，提升多机协作效率。在密集任务（多机竞争同一目标）场景下效果最显著。代价是每 batch 增加一次 LLM 调用，需权衡协作质量与调用成本。

11.2 自然语言通信替代数值向量（简化通信架构）

问题诊断：56+ 消息类型的维护负担

当前 CommunicationEnums.cs 中定义了 56+ 种消息类型，每种类型对应固定结构体和解析逻辑。问题在于：

  • 新增协作模式需要新增消息类型 + 解析代码，扩展成本高
  • 非约束性消息（"我准备好了"、"需要支援"）用结构体描述信息损失大
  • LLM 无法直接"读取"结构体消息，需要额外序列化注入 prompt
  • 消息语义僵化，无法表达上下文依赖的灵活意图

RALLY 的方案：自然语言替代数值向量

RALLY 用自然语言字符串替代传统的数值消息向量。智能体广播：

"UAV-1 已到达北门区域，正在执行侦察，预计完成时间 45 秒，当前电量 82%"

而非结构体：{type: StatusUpdate, agentId: "UAV-1", location: [x,y], battery: 0.82, eta: 45}

接收方 LLM 直接理解语义并在决策时使用，无需解析代码。这也是 RALLY 实现带宽压缩的核心机制之一。

分层改造策略：不是全部改，而是按类型分层

关键原则：精确约束信号保留结构体（正确性优先），非约束性协作意图改为自然语言（灵活性优先）。

消息类别策略理由示例
C1/C2/C3 约束信号（互斥、同步、等待） 保留结构体 高频、低延迟、精确性要求高，不容语义歧义 DoneSignal / MutexRequest
槽位认领/释放 保留结构体 Slot 系统的核心信号，需要原子性保证 SlotClaim / SlotRelease
紧急事件广播 改为 NL 事件语义复杂，NL 表达更完整，LLM 可直接推断响应策略 "UAV-2 在东南角发现可疑人员，建议北侧智能体支援"
状态共享/意图广播 改为 NL 非实时信号，语义丰富度比结构体精度更重要 "UAV-1 完成北区巡逻，准备返回基地充电，任务可移交"
MAD 辩论发言 已是 NL（保持） 辩论本就是自然语言，复用此通道即可 辩论 JSON 中的 argument 字段
非约束性协作请求 改为 NL 新增协作模式无需修改枚举定义，直接广播意图文本 "需要一名智能体协助覆盖西侧感知盲区，持续约 60 秒"

报告总结

本报告通过深度分析当前 UnityLLM 多智能体系统与斯坦福小镇范式的差距，提出了一套系统性的架构演进方案。核心结论：

  1. 分组和Slot系统是正确且必要的（本次更新核心结论）。无人机需要物理协作（时序严格、方向互斥），不能像斯坦福小镇那样让每个agent完全独立涌现。Slot是「防止协作崩溃」的基础，而非束缚涌现的枷锁。涌现应发生在执行层，不是角色层。
  2. 最大的改进点是去掉 LLM#4（固定步骤预生成），将 LLM#5（ADM）升级为主决策引擎：输入 = Slot角色 + 实时感知 + 白板状态 + 记忆上下文，每个决策周期动态生成下一个动作。这是实现「执行层涌现」的核心。
  3. 记忆模块需从"结构化存储"升级为带有三维评分（时效×重要性×相关性）的语义检索系统，作为ADM响应式决策的上下文燃料，也驱动Slot认领时的角色偏好。
  4. 反思模块需从单层失败触发升级为基于重要性累积的层级递归反思（L1观察→L2反思→L3抽象概念）。无人机场景中，反思结果不仅写入个人记忆，还可以提取到SkillLibrary供所有agent复用。
  5. 新增模块优先级：① 自主目标生成器（实现无人工干预自治）② 社会关系图谱（驱动自发搭档关系涌现）③ 技能库（积累协调经验、减少重复学习）④ 人格系统（角色偏好分化的基础）。
  6. 现有系统的 MAD 辩论、分组+Slot、C1/C2/C3约束协调、A*导航能力已超过斯坦福小镇的对应模块，是宝贵的竞争优势，无需重构，只需扩展与增强。

12.3 MAD 期间 Agents 如何通信——完整消息流

5 种 MessageType 构成完整的 MAD 通信协议，全部通过现有 CommunicationModule 路由，无需额外网络层。

📤 IncidentReport（成员 → Leader）
触发入口。任意成员（通过 DebateGateway 或手动构建）发送给 Leader。Leader 侧 CommunicationModule 路由给 GroupMonitor.HandleIncidentReport()。
Payload：reporterId / affectedAgentId / incidentType / description
📢 IncidentAnnounce（Leader → 全员）
Phase 0 广播。Leader 生成 incidentId 后写白板（IncidentAnnounce 类型）并消息广播。Critical 事件时，接收方 DebateParticipant.OnCriticalIncident() 立即中断当前 ActionBatch。
🎭 DebateProposal（Leader → 各成员，点对点）
每轮开始时 IncidentCoordinator 为每位成员分配角色。Round 1：受影响Agent+Leader=Proposer，其余=Critic。Round 2+：全员=Voter。
Payload：incidentId / leaderId / report / role / debateRound / existingEntriesSummary
🗣️ DebateUpdate（成员 → Leader）
成员在 Rolling Loop Step 11.5 窗口响应。调用 LLM 生成 DebateEntry（content / confidence / voteFor），发回 Leader。IncidentCoordinator.AddDebateEntry() 汇总。
⚖️ DebateResolved（Leader → 全员）
收敛判断通过后（多数投票 OR 置信集中 OR 轮次上限）广播。写 DebateConsensus 到白板，各成员 ADM.OnDebateResolved() 接收并执行共识。若 missionScopeChanged=true，触发 RequestReplan()。

Agent 掉线容错：FREE-MAD 思路（Cui et al. arXiv 2025）

当前问题：IncidentCoordinator 等待全员回复，成员掉线时会卡在超时（12-18s）。FREE-MAD 不需要全员共识——基于已有辩论轨迹的置信度加权做决策：

// 在 IncidentCoordinator 收到部分回复后即可调用（无需等满员）

private DebateConsensusEntry ResolveWithPartialEntries(List<DebateEntry> received)
{
    // 对已收到的 Proposer 按 confidence 加权
    var voteWeights = received
        .Where(e => e.role == DebateRole.Voter && !string.IsNullOrEmpty(e.voteFor))
        .GroupBy(e => e.voteFor)
        .ToDictionary(g => g.Key, g => g.Sum(e => e.confidence));

    // 最终得分 = 自身 confidence + 获得的投票加权总和
    var best = received
        .Where(e => e.role == DebateRole.Proposer)
        .OrderByDescending(p => p.confidence + voteWeights.GetValueOrDefault(p.entryId, 0f))
        .FirstOrDefault();

    return new DebateConsensusEntry
    {
        incidentId = _activeIncident.incidentId,
        resolution = best?.content ?? "无有效提案，执行规则仲裁",
        decidedAt  = Time.time
    };
}

12.4 资源受限下的 MAD——三层降级策略

研究报告指出：应建立统一资源预算模型，将通信 token + 计算延迟 + 电量消耗合并为效用函数，在辩论前评估"这次辩论值不值得"。这是 LLM-MAS 文献中几乎没有的研究角度。

注：三层均可包含 LLM 调用，区别在于调用次数和轮次。Layer 3 是唯一的多轮辩论层；Layer 1/2 最多各进行一次 LLM 调用，用于验证或综合，不构成辩论。当前系统默认全部走 Layer 3（即 DebateLayer.FullLLM），三层降级策略留作后期性能优化。

层次触发条件机制LLM 调用次数token 开销延迟
Layer 1：规则优先仲裁 电量 < 30% OR urgency > 0.9 OR 冲突复杂度 = 1 确定性规则优先（电量最高/距离最近/任务权重最高者优先）。规则无法定论时，可选发起 1 次 LLM 验证调用（生成简短结论句），不做多轮交互。 0 ~ 1 次 0 ~ 200 <1ms（规则）/ <2s（+LLM）
Layer 2：JSON 摘要 + 单次 LLM 综合 冲突复杂度 ≤ 2 AND 电量 30%~50% 各 Agent 上报 StructuredArgument JSON（量化摘要，无自然语言辩论）。Leader 收集所有摘要后，发起 1 次 LLM 调用进行综合分配并返回决策，不做多轮交互。 1 次 300 ~ 600 1 轮通信 + 1 次 LLM
Layer 3：完整 LLM MAD 资源充裕 AND 冲突复杂度高（当前默认路径） 完整多轮辩论：自然语言 Proposer/Critic/Voter，每轮各角色独立 LLM 调用，Arbiter 仲裁收敛。 N × 角色数（多轮） 500 ~ 2000 /轮 1 ~ 3s /轮

Unity 单线程下 C3-1 的实际工作方式

C3-1 不存在"所有 agent 同时决策再检冲突"的场景。互斥锁（SharedWhiteboard._mutexLocks）将决策过程串行化：同一时刻只有一个 agent 持锁并执行 Step 0→6（读白板 → LLM → 写占位）。后来的 agent 读到白板中已有的 IntentAnnounce.progress=targetName，LLM 主动避开 → 语义层无冲突。这是先到先得 + LLM 规避机制，而非冲突检测。

MAD 触发的是异常场景，而不是正常竞争。

两个真正需要 MAD 的场景：

场景 A：持锁方卡死——等待超时（AcquireC3MutexLocks 内部）

等待方循环重试 >5s，说明持锁方 LLM 超时或执行阻塞，正常重试无法自愈，触发 MAD 让全组讨论是否强制转让目标。

// ActionDecisionModule.cs — AcquireC3MutexLocks() 内部修改

private const float C3MutexDebateThreshold = 5f; // 新增常量：等待超时阈值
private bool _c3DebateTriggered = false;          // 新增字段：每轮迭代进入 Step 0 时重置

private IEnumerator AcquireC3MutexLocks(List<string> constraintIds)
{
    if (constraintIds == null || constraintIds.Count == 0) yield break;
    string myId = agentProperties?.AgentID ?? string.Empty;
    float elapsed = 0f;

    while (elapsed < 60f)
    {
        bool allAcquired = true;
        foreach (var cid in constraintIds)
            if (!SharedWhiteboard.Instance.TryAcquireMutexLock(cid, myId))
            { allAcquired = false; break; }

        if (allAcquired) { _c3DebateTriggered = false; yield break; }

        foreach (var cid in constraintIds)
            SharedWhiteboard.Instance.ReleaseMutexLock(cid, myId);

        // ── 新增：等待超过阈值，持锁方疑似卡死，触发 MAD ──────────────────
        if (elapsed > C3MutexDebateThreshold && _madGateway != null && !_c3DebateTriggered)
        {
            _c3DebateTriggered = true;
            // GetMutexHolder() 为 SharedWhiteboard 新增的只读查询（约 3 行）
            string holder = SharedWhiteboard.Instance.GetMutexHolder(constraintIds[0]);
            _madGateway.Raise(new DebateRequest
            {
                initiatorId     = myId,
                affectedAgentId = holder,          // 持锁方（疑似卡死）
                incidentType    = IncidentType.AgentImpaired,
                topic           = $"C3 互斥锁长期阻塞: {constraintIds[0]}",
                context         = $"{myId} 等待互斥锁 {constraintIds[0]} 已 {elapsed:F0}s，"
                                + $"持锁方 {holder} 疑似卡死，请团队讨论是否强制转让目标。",
                estimatedRounds = 1
            });
        }
        // ───────────────────────────────────────────────────────────────────

        yield return new WaitForSeconds(0.1f);
        elapsed += 0.1f;
    }
    Debug.LogError($"[ADM] {myId} 获取 C3-1 互斥锁超时！强制继续");
}

十三、自主涌现系统重设计

本次重设计聚焦三个核心问题：① 所有 agent 同时 Idle 时同时触发涌现，没有空闲 agent 可接受邀请； ② 独立任务走完整 LLM1-4 流水线，LLM2/3 对单人任务无意义； ③ 协作是事前规划出来的，不是执行中自然发生的。 解决方案：两条完全独立的执行路径 + 感知事件触发协作 + 人格驱动的对抗型 agent。

13.1 双路径架构总览

设计原则：两条路径完全独立，不删任何现有代码。
外部任务路径（LLM1→4）原封不动；自主涌现路径新增，绕过 LLM1/2/3。

外部任务路径（不变）

自主涌现路径（新增）

Solo：

13.2 Solo 涌现路径

每个 agent 基于自身内驱力完全独立评估，不等待任何队友。5 个驱动维度：

Battery

D = 1 − battery%

电量耗尽时最强，触发充电类任务。

Threat

D = recency×0.6 + density×0.4

结合记忆中敌方目击频率和时效，受 neuroticism 调制。

Exploration

D = 1 − uniqueTargets/10

已覆盖区域越少越强，受 openness 调制。

Collaboration

D = timeSinceCollab/300s

距上次协作越久越强，受 agreeableness 调制。

Disruption（新增）

D = activeEnemies × 0.5

仅对抗型激活。感知到越多敌方 active 任务，干扰动机越强。

触发条件

topDrive > 0.6
+ jitter 0-5s

随机 jitter 降低多 agent 同步触发概率。

关键改变：EmergenceResult 字段从 needsHelp: bool 改为 steps: string[]。 Solo 任务直接携带 LLM 生成的步骤注入 PlanningModule，不再发起任何广播，彻底消除"所有人同时 Idle 时无人接受邀请"的死锁。

13.3 感知触发协作路径

协作不在任务开始前规划，而在 Solo 执行途中感知到机会或威胁时自然触发——这才是真正的涌现。

人格兼容性检查：在 OnColabInvite() 入口， initiatorPs.IsAdversarial != _personalitySystem.IsAdversarial 为 true 时直接 return， 不消耗任何 LLM token。协作型与破坏型天然隔离，互不干扰对方的涌现协作链路。

ComputeDisruptionDrive() — 仅对抗型激活
D = Clamp01( NactiveEnemies × 0.5 )
