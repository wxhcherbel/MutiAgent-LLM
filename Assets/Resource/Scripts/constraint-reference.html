StructuredConstraint

协同约束 · 通用字段

每条约束无论类型如何，都必须填写这四个通用字段。
约束由 LLM#1 从自然语言中提取，存储在 ParsedMission.constraints[] 中， 后续各阶段通过 constraintId 引用，不重复存储内容。

constraintId string 约束的全局唯一编号

在本次任务内唯一标识一条约束。格式固定为 "c_NNN"，由 LLM#1 生成时按序分配。
PlanSlot 和 PlanStep 中的 constraintIds[] 引用此字段， ADM 通过此 ID 查询白板或监听裁判事件，不需要重复复制约束内容。

举例
// 搬家任务中的同步起抬约束
constraintId = "c_003"

// PlanStep 引用此约束（只存ID，不存内容）
step.constraintIds = ["c_001", "c_003"]
cType string 约束类型，决定使用哪组专用字段

四个值之一，对应四类约束。填写后，ADM 只读取该类型对应的专用字段，其余字段忽略。

"Assignment" C1 分配约束：谁负责什么目标/资源，是否互斥
"Completion" C2 完成条件：任务达到什么状态算完成，是否需要多机同步
"Coupling" C3 耦合约束：我的行为和其他智能体如何相互影响（合作/竞争/对抗）
"Priority" C4 优先级约束：多个约束冲突时如何裁决，触发后执行降级行为
channel string 约束状态通过什么机制传递

决定 ADM 执行此约束时去哪里读取状态更新：组内白板，还是等裁判宣布。

"whiteboard" 组内白板，组员互相可见。适用于合作任务和对抗任务的己方内部协同。ADM 主动轮询白板中对应 constraintId 的条目。
"referee" 等待裁判宣布物理事实。适用于竞争任务（谁先到达）和对抗任务的跨组信息（敌方状态）。ADM 被动监听裁判广播。
规律： C3 sign=+1（合作）→ whiteboard； C3 sign=0（竞争）或 -1（对抗）→ referee； C1/C2 组内 → whiteboard；涉及跨组资源 → referee
involvedAgents string[] 此约束涉及哪些智能体

填写受此约束影响的 agentId 列表。空数组表示约束适用于本组全体成员。 ADM 在响应约束时，只观察此列表中的成员状态，不关心列表外的其他智能体。

举例
// C2同步完成约束：三架都要完成才算任务结束
involvedAgents = ["drone_A", "drone_B", "drone_C"]

// C1分配约束：只约束 drone_A 负责某充电桩
involvedAgents = ["drone_A"]

// 空数组：本组全体都受此约束
involvedAgents = []
C1 · Assignment

分配约束

规定"谁负责什么"。防止任务被遗漏（没人管）或资源被重复占用（两架挤一个桩）。
合作任务中用于分工，竞争任务中用于声明占有，对抗任务中用于分配包抄方向。

subject string 谁来负责执行

执行此分配任务的智能体 ID 或角色名。填 "all" 表示本组全员共同负责同一目标（如合抬重物）。

举例
// 合作搬家：全员共同负责沙发
subject = "all"

// 竞争抢桩：drone_A 声明自己去占桩1
subject = "drone_A"

// 对抗包抄：red_A 负责东侧方向
subject = "red_A"
targetObject string 被分配的目标对象

可以是地点名、资源名、任务名或方向描述。LLM#4 拆解步骤时读取此字段生成具体动作。

举例
// 合作
targetObject = "沙发"

// 竞争
targetObject = "充电桩_1"

// 对抗
targetObject = "北侧包抄路线"
exclusive bool 同一目标是否只能被一个智能体占有

true：互斥，同一时刻只有一个智能体可以持有此目标。裁判宣布占有后，其他智能体必须放弃并重新选择。
false：共享，多个智能体可以同时作用于同一目标（如合抬同一件重物）。

举例
// 竞争抢桩：一个桩只能一架占
exclusive = true

// 合作合抬：大家都作用于同一个沙发
exclusive = false

// 对抗包抄方向：两架不能堵同一方向，否则另一侧空缺
exclusive = true
C2 · Completion

完成条件约束

规定"做到什么程度算完成"。支持单机独立完成和多机同步完成两种模式。
channel=whiteboard 时，各成员写 DoneSignal，等所有人写完才触发；channel=referee 时，由裁判直接宣布结果。

condition string 完成判定标准（自然语言）

描述任务完成的客观状态，供 ADM 判断当前步骤是否可以标记为完成。 尽量写成可被感知传感器或裁判观察验证的客观描述，避免主观判断。

举例
// 合作搬家
condition = "所有家具已到达新家指定位置"

// 竞争抢桩
condition = "充电头已插入充电桩且电量开始上升"

// 对抗红方
condition = "蓝方进入以集结点为圆心的50米包围圈"

// 对抗蓝方
condition = "存活时间超过300秒"
syncWith string[] 需要等哪些成员也完成才算整体结束

非空时：ADM 完成自身条件后，还需等 syncWith 中所有成员都在白板写入 DoneSignal，才推进到下一步。
空数组时：只需自己完成，不等其他人。
注意：channel="referee" 时此字段无效，由裁判直接宣布整体结果。

举例
// 合作：三架都到达新家才算完成
syncWith = ["drone_B", "drone_C"]
// drone_A 完成后在白板写 DoneSignal，等 B 和 C 也写了再结束

// 竞争：个人任务，不等其他人
syncWith = []

// 对抗：裁判宣布，syncWith 不填
channel   = "referee"
syncWith  = []  // 无效，裁判负责
C3 · Coupling

耦合约束

规定"我的行为和其他智能体如何相互影响"。这是三类任务本质的区别所在：结构相同，符号不同。
合作=+1（等对方就位），竞争=0（对方占了我换目标），对抗=-1（对方逼近我立即响应）。

sign int 耦合方向，三类任务的核心区别

对方的行为对我的影响性质。只填三个值。

+1 合作 对方行动有利于我。我需要等对方就位才能继续行动。
举例：搬沙发时等另两架都到抬起位置才起飞
0 竞争 对方抢走资源使我的选择减少，但不主动攻击我。我感知到对方占用后重新选择目标。
举例：对方先到充电桩1，我转向充电桩2
-1 对抗 对方主动以我为目标，我需要实时响应对方动作。感知到威胁立即调整行为，不等待。
举例：红方无人机逼近，蓝方立即启动规避机动
watchAgent string 我需要观察谁的状态

指定 ADM 去哪里读取状态更新。与 channel 字段配合使用：
channel=whiteboard 时，填具体的 agentId，ADM 在白板中过滤该 agent 的条目；
channel=referee 时，填 "referee"，ADM 监听裁判广播。

举例
// 合作 sign=+1：观察队友白板
channel     = "whiteboard"
watchAgent  = "drone_B"

// 竞争 sign=0：看不到对方白板，等裁判宣布资源状态
channel     = "referee"
watchAgent  = "referee"

// 对抗 sign=-1：看不到敌方白板，靠裁判广播敌方位置
channel     = "referee"
watchAgent  = "referee"
reactTo string 触发我响应的事件描述

自然语言描述，ADM 将此字符串与 WhiteboardEntry.status 或 RefereeEvent.payload 做语义匹配， 判断是否满足触发条件。匹配成功则执行对应行为（等待/重新选目标/规避）。

举例
// 合作：等另外两架都报告就位
reactTo = "drone_B 已到达抬起位置"

// 竞争：裁判宣布桩被占，我换目标
reactTo = "充电桩_1 已被占用"

// 对抗：裁判宣布红方进入我的威胁范围
reactTo = "红方无人机进入我方100米范围内"
C4 · Priority

优先级约束

规定"约束冲突时怎么裁决"。当电量不足、禁飞区、任务超时等紧急情况出现时， C4 覆盖其他约束，让 ADM 执行降级行为而不是继续强行推进原计划。
ADM 在每个步骤执行前都会检查本步骤绑定的所有 C4 约束的 triggerCondition。

overrides string[] 触发时使哪些约束暂时失效

填 constraintId 列表。C4 触发后，ADM 忽略列表中所有约束的要求，直接执行 fallback 行为。 任务恢复正常后，被覆盖的约束重新生效。

举例
// drone_B 电量不足，覆盖"必须三架同时起抬"的要求
overrides = ["c_003"]
// c_003 是 C3 耦合约束（sign=+1，等三架同时就位）
// 覆盖后另两架不必再等 drone_B

// 禁飞区警告，覆盖追捕目标和包抄分工
overrides = ["c_001", "c_003"]
triggerCondition string 触发此优先级约束的条件

自然语言描述，ADM 在每步执行前检查此条件是否成立。 可以是传感器读数（电量）、位置判断（禁飞区）、时间（超时）或白板/裁判事件。

举例
// 电量触发
triggerCondition = "任意成员电量低于 10%"

// 位置触发
triggerCondition = "我方任意无人机飞入禁飞区边界 50 米内"

// 时间触发
triggerCondition = "任务已超时，剩余时间为 0"
fallback string 触发后执行的降级行为

自然语言描述，C4 触发时 ADM 放弃当前步骤，改为执行此行为。 应足够具体，让 LLM#5 能直接生成原子动作，不需要再次调用 LLM#4 重新规划。

举例
// 电量不足降级
fallback = "drone_B 立即返回最近充电站，其余成员继续执行原计划"

// 禁飞区降级
fallback = "全体立即原地悬停，等待新指令，禁止继续向当前目标移动"

// 超时降级
fallback = "放弃当前目标，返回出发点待命"
WhiteboardEntry

组内白板条目

白板是组内共享的状态池，每个组员都可以读写。
ADM 执行步骤前轮询白板，通过 constraintId 过滤出与本步骤相关的条目， 将 status 与 C3 的 reactTo 做匹配，决定是否等待或继续。
注意：白板只在组内可见，对方组和裁判无法读取。

agentId string 谁写了这条记录

写入方的智能体 ID。其他成员通过此字段确认"我在等的那个人"是否已经更新状态。

举例
agentId = "drone_B"
// drone_A 在白板查询时过滤 agentId="drone_B" 的条目
timestamp float 写入时的仿真时间（秒）

ADM 可通过此字段判断信息是否过期。超过一定时间未更新的条目应视为失效，触发重新等待或 C4 降级。

举例
timestamp = 42.5
// 任务开始后第 42.5 秒写入
// 若当前时间为 52.5，超过 10 秒未更新，视为超时
constraintId string 此条记录属于哪条约束

ADM 不读取白板上全部内容，只读取与当前步骤的 constraintIds[] 匹配的条目。 这样每个 agent 只关心自己关心的约束，避免信息过载。

举例
// drone_A 当前步骤绑定的约束：["c_002", "c_003"]
// ADM 只在白板查询 constraintId = "c_003" 的条目
constraintId = "c_003"
entryType string 这条记录的语义类型

ADM 根据 entryType 决定如何响应此条目。

"IntentAnnounce" 意图宣告：我即将执行某动作，提前告知他人避免冲突。
举例：drone_A 宣告"我准备从东侧接近目标，请队友避让"
"ReadySignal" 就绪信号：我已完成前置动作，等待同步触发。C3 sign=+1 时使用。
举例：drone_A 宣告"已到达沙发抬起位置，等待同步"
"DoneSignal" 完成信号：我已满足本约束要求的条件。C2 syncWith 检查时使用。
举例：drone_A 宣告"沙发已放置到新家指定位置"
"StatusUpdate" 周期性状态汇报，供队友参考。通常用于 C4 触发前的预警。
举例：drone_B 汇报"当前电量 8%，即将触发 C4 降级约束"
status string 状态的自然语言描述

其他成员的 ADM 将此字段与 C3 的 reactTo 做语义匹配，判断是否满足触发条件。 因此措辞应尽量与 reactTo 的描述一致，方便匹配。

举例（与 reactTo 对应）
// C3 reactTo = "drone_B 已到达抬起位置"
// drone_B 白板写入：
status = "已到达沙发抬起位置，等待其他成员就位"
// ADM 语义匹配成功 → drone_A 解除等待，继续执行
progress float 完成进度 0~1，不需要时填 -1

主要供 C2 同步完成约束使用：组长或协调者可通过轮询所有成员的 progress 判断整体进度， 决定是否等待或触发超时 C4。不涉及进度跟踪的约束填 -1。

举例
// drone_A 正在飞往新家途中，完成了 75% 的路程
progress = 0.75

// 到达新家，任务完成
progress = 1.0

// 纯意图宣告，不需要进度
progress = -1
RefereeEvent

裁判事件

裁判负责观察物理世界中真实发生的事，宣布客观结果。
裁判不做分配、不发命令、不做决策——只宣布事实。
各智能体收到裁判事件后，由自身 ADM 根据 C3 的 reactTo 自行决定如何响应。
类比足球裁判：只吹哨宣布越位，不告诉球员该往哪跑。

eventType string 裁判宣布的事件类型
"ResourceClaimed" 某互斥资源已被某智能体物理占有（不是声明，是真实到达）。
举例：drone_A 先物理接触充电桩1并开始充电
"ResourceReleased" 之前被占的资源重新可用。
举例：drone_A 充电完成，离开充电桩1
"AgentEliminated" 某智能体在对抗中被淘汰，退出任务。
举例：red_A 被蓝方反制，不再参与任务
"ConditionMet" 某个 C2 完成条件被裁判判定为满足（跨组任务使用）。
举例：红方成功将蓝方逼入50米包围圈
"MissionResult" 整个任务的最终胜负宣布，所有智能体停止执行。
举例：蓝方存活超过300秒，蓝方逃脱成功
"PriorityOverride" 裁判触发全局 C4 约束（如禁飞区），所有相关智能体必须暂停当前行动。
举例：检测到无人机飞入禁飞区边界
constraintId string 触发此事件的约束 ID

ADM 通过此字段判断此裁判事件是否与自己当前步骤的约束相关。 不相关的事件直接忽略，不影响当前执行流程。 无对应约束时（如最终结果宣布）填空字符串。

举例
// 充电桩互斥约束被触发
constraintId = "c_001"

// 最终任务结果，不对应具体约束
constraintId = ""
payload string 裁判宣布的客观事实描述

自然语言描述。ADM 将此字段与 C3 的 reactTo 做语义匹配， 决定是否需要调整行为。因此措辞应尽量明确，不含主观判断。

举例
// ResourceClaimed：竞争抢桩
payload = "充电桩_1 已被 drone_A 物理占用，其余智能体不可使用"

// AgentEliminated：对抗淘汰
payload = "red_A 被蓝方反制，已退出任务"

// MissionResult：最终结果
payload = "蓝方无人机存活超过 300 秒，蓝方逃脱成功，任务结束"

// PriorityOverride：禁飞区
payload = "drone_C 进入禁飞区边界 50 米内，全体暂停当前任务原地悬停"
visibleTo string[] 哪些智能体可以收到此事件

裁判控制信息可见性，实现对抗任务中的信息不对称。 同一个物理事件，裁判可以选择只告知部分智能体，模拟真实战场的情报差异。

举例
// 竞争：充电桩状态所有人都能看到
visibleTo = ["drone_A", "drone_B", "drone_C"]

// 对抗：红方位置只告诉红方自己（蓝方不知道红方在哪）
visibleTo = ["red_A", "red_B"]

// 对抗：蓝方逃脱成功，全体都能知道结果
visibleTo = ["red_A", "red_B", "blue"]
timestamp float 裁判宣布时的仿真时间（秒）

ADM 可用于计算响应延迟、判断事件是否过期，或记录比赛日志。

举例
timestamp = 47.3
// 任务开始后第 47.3 秒，drone_A 物理到达充电桩1
// 裁判在此刻宣布 ResourceClaimed 事件
参考

三类任务对比

四类约束在三种任务中的字段填法规律。结构完全相同，只有参数和 sign 不同。

字段 合作（搬家） 竞争（抢充电桩） 对抗（猫鼠游戏）
channel whiteboard referee（桩状态） whiteboard（己方内部）
referee（敌方信息）
exclusive false（合抬同一物） true（桩不能共享） true（方向不能重叠）
syncWith 全组成员（三架都到才完成） 空（各自完成各自的） 空（裁判宣布，此字段无效）
sign +1（等队友就位） 0（对方占了我换目标） +1（己方内部合作）
-1（对敌方响应）
watchAgent 具体 agentId（看队友白板） "referee"（等裁判宣布） agentId（看队友）
"referee"（看敌方）
白板用途 同步就绪信号 + 完成信号 几乎不用 己方内部同步包抄方向
裁判介入时机 仅 C4 触发时 全程（宣布资源占用状态） 全程（宣布对抗结果 + 最终胜负）
