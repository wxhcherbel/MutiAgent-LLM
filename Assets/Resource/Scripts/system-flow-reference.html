Code Baseline First

当前项目不是“按模块平铺”的系统，而是一条被多处状态机穿起来的主链

真实入口不是 Web 面板，也不是某个“中央裁判脚本”，而是 PlanningModule.SubmitMissionRequest(string missionDescription, int agentCount)。 该入口先让一个智能体完成任务解析与分组，再通过消息广播把组信息发给所有成员。随后每个成员各自拿到 PlanSlot、拆解为 PlanStep，再由 ActionDecisionModule 在滚动循环里把步骤解释成一批批 AtomicAction[]。白板、通信、裁判、感知、运动执行都不是独立主线， 而是在这条链的不同节点上插入上下文、约束和反馈。

真实入口 PlanningModule.SubmitMissionRequest() + IntelligentAgent.CheckForDecision()
规划核心 LLM#1 解析任务，LLM#2 产槽，LLM#3 选槽，LLM#4 拆步骤
执行核心 RunRollingLoop() 负责把步骤滚动转成 AtomicAction[]
审阅原则 代码事实优先，旧设计稿偏差只做旁注，不把推测写成现状
阶段 0 · 调度入口

谁来触发规划：不是一直在跑 LLM，而是 IntelligentAgent 轮询后才进入主链

项目中每个智能体都有自己的 IntelligentAgent 控制器。 它在 Update() 里调用 CheckForDecision()，每隔 2 秒检查一次是否需要决策。 如果当前已经有活动任务且 Agent 处于空闲态，则进入 MakeDecisionCoroutine()，再调用 ActionDecisionModule.DecideNextAction()。 如果是外部任务输入，则由任意一个空闲的 PlanningModule 收到 SubmitMissionRequest()，把系统带入规划态。

IntelligentAgent.Update() CheckForDecision() MakeDecisionCoroutine() PlanningModule.SubmitMissionRequest()

输入

  • 用户任务文本 missionDescription
  • 参与智能体数量 agentCount
  • Agent 当前状态 CurrentState.Status
  • 本地轮询节拍 decisionCheckInterval = 2.0f

输出

  • 规划入口被打开，busy = true
  • PlanningState 从 Idle 进入 Parsing
  • 已有活动任务时，进入 ADM 决策而不是重新发起任务

关键字段

  • isMakingDecision：防止同一智能体重复进入决策协程
  • lastDecisionTime：控制轮询节拍
  • busy：防止同一 PlanningModule 并发提交任务

涉及函数

  • IntelligentAgent.CheckForDecision()
  • IntelligentAgent.ShouldMakeDecision()
  • IntelligentAgent.MakeDecisionCoroutine()
  • PlanningModule.SubmitMissionRequest()

上下游关系

  • 上游：UI、HTTP 控制台、脚本直接调用
  • 本阶段只负责“是否进入规划”
  • 下游直接进入 RunLLM1()

调用链引用

AgentStateServer.ExecuteCommand() 或本地脚本 -> PlanningModule.SubmitMissionRequest() -> StartCoroutine(RunLLM1())。 如果当前不是新任务，而是已有活动步骤继续推进，则是 IntelligentAgent.MakeDecisionCoroutine() -> ActionDecisionModule.DecideNextAction()。

调度表现
先由一个空闲 PlanningModule 受理任务；后续每个成员的决策由各自 Agent 定时驱动。
通信含义
此阶段还没有队内消息，只是把任务送进规划入口。

用户输入

调度表现
入口和合作任务相同，但后续应当更依赖裁判事件而不是纯白板协同。

对抗

红队围堵蓝队逃脱

用户输入
“红队拦截蓝队，蓝队尝试从教学楼东侧撤离。”
调度表现
入口同样统一；差异不在入口，而在后续分组、目标冲突和裁判反馈。
主链提示
这类任务需要队内协同和队间对抗并存，因此后续会同时出现白板与裁判话题。
已接入 入口调度本身比较清晰：新任务和已有任务推进是两条不同入口 v

这里最容易误解的一点是：系统并不是“用户发起一次请求后，所有 Agent 同时跟着跑一条大协程”。 实际上只有任务解析阶段是某个 PlanningModule 主导，后续每个 Agent 的动作推进都由自己的 IntelligentAgent 周期性触发。这个设计让局部行动具备自治性，也解释了为什么 ADM、感知、运动执行都挂在每个 Agent 身上。

优化建议 多 Agent 框架论文提示：入口后应尽快进入 SOP 化协作，而不是只靠自然语言自由发挥 v

AutoGen、CAMEL、MetaGPT 这类工作都强调：多 Agent 一旦进入复杂任务，必须尽快把角色边界、 交互格式、可共享中间结果标准化，否则后续链路会出现级联幻觉。

  • AutoGen 适合补“多 Agent 会话协议”和消息格式校验。
  • CAMEL 适合补“角色提示词”和角色一致性。
  • MetaGPT 适合补“标准作业流程 SOP”与发布-订阅式共享消息池。
推荐插入点：任务刚进入规划前，把“允许产出的结构”“角色边界”“失败回退格式”写成统一系统协议。
阶段 1 · LLM#1 任务解析

RunLLM1() 把用户自然语言压成 ParsedMission，这是后续所有协同约束的源头

PlanningModule.RunLLM1(string desc, int cnt) 会构造一个非常长的提示词， 要求 LLM 识别任务关系类型 relType、分组数 groupCnt、组任务 groupMsns、时间限制 timeLimit，以及最关键的 StructuredConstraint[] constraints。当前代码里真正落地的协同约束只有 C1 / C2 / C3 三类。

PlanningState.Parsing LLMInterface.SendRequest() ParsedMission StructuredConstraint

输入

  • 用户任务文本 desc
  • 参与智能体数 cnt
  • LLM 提示中内建的 C1/C2/C3 规则与示例

输出

  • ParsedMission.msnId
  • relType / groupCnt / groupMsns
  • timeLimit
  • constraints[]

关键字段

  • relType：合作、竞争、对抗或混合
  • groupScope：约束属于哪一组
  • watchAgent / syncWith：供运行期补全和查询

涉及函数

  • PlanningModule.RunLLM1()
  • LLMInterface.SendRequest()
  • PlanningModule.ExtractJson()
  • PlanningModule.AssignGroups()

上下游关系

  • 上游：入口已确认可提交任务
  • 本阶段产出的是“全局任务图”而非个人计划
  • 下游立即进入本地分组与组内广播

调用链引用

PlanningModule.SubmitMissionRequest() -> RunLLM1() -> LLMInterface.SendRequest(prompt, callback) -> JsonConvert.DeserializeObject<ParsedMission>() -> AssignGroups(parsedMission, agentIds)。

字段内嵌 首次出现的数据结构：ParsedMission 与 StructuredConstraint v
ParsedMission.msnId 任务唯一 ID，由 GenMsnId() 生成，后续所有消息和步骤都用它隔离任务轮次。
ParsedMission.relType 任务关系类型。会影响分组数量与后续协同解释，但当前代码没有专门按类型切不同执行器。
ParsedMission.constraints 全局约束集合。之后不会重复拷贝内容，只会在槽位与步骤里通过 constraintId 引用。
StructuredConstraint.cType 当前代码真实支持的是 C1、C2、C3。没有 C4 Priority 实装字段。
StructuredConstraint.channel 当前主链会用到 whiteboard 与少量 referee 读取语义，但裁判实际广播链不完整。 
// LLM#1 期望的核心 JSON 轮廓：这里只展示字段意图，不贴原始源码
{
  "relType": "Cooperation",
  "groupCnt": 1,
  "groupMsns": ["东区侦察 + 西区侦察 + 同步回传"],
  "timeLimit": 0,
  "constraints": [
    // C1：谁负责什么

审阅风险 LLMInterface 已进入主链，而且当前代码里直接写了 API Key v

当前 LLM_API/LLMInterface.cs 的默认配置中直接包含兼容 OpenAI Chat Completions 的 apiUrl、apiKey 和默认模型。这意味着任何拿到仓库副本的人都能看到密钥。 这不是文档推断，而是已经存在于主链的真实安全问题，因为所有 LLM#1/2/3/4 和 ADM 滚动规划都经过它。

  • 风险级别：高。因为密钥泄露会立即影响费用与账号安全。
  • 最小正确改法：移到环境变量、Unity Secret、外部配置文件或本地未入库资源。
  • 文档层处理：本页只做风险标识，不默认替你改运行时代码。

对抗

输入如何变成对抗约束

用户文本
“红队围堵蓝队，蓝队逃离东门。”
LLM#1 输出重点
应识别为 Adversarial，至少拆成两组任务，并把组内协同约束通过 groupScope 限定在各自组内。
主链含义
这一层先把“红蓝对立”定下来，但对手信息后续是否进入决策，还依赖裁判链是否真的广播。
设计偏差 旧 constraint-reference.html 中的 C4 / involvedAgents 不是当前代码事实 v

仓库里已有的 constraint-reference.html 更像一版更激进的设计草稿，里面出现了 C4 Priority、involvedAgents、更完整的裁判机制等概念。 但当前 PlanContracts.cs 真实定义的结构只有： constraintId、cType、channel、groupScope、 subject、targetObject、exclusive、 condition、syncWith、sign、 watchAgent、reactTo。

所以本页从这里开始就只按代码支持的字段往下讲，不把设计稿字段混进主链解释。
优化建议 多 Agent 任务描述可以参考 MetaGPT 的 SOP 思路，让 LLM#1 输出更稳定 v

当前 LLM#1 依靠长提示词硬编码三类约束识别规则，能跑，但对复杂任务容易变脆。 MetaGPT 的价值不在“角色名更好看”，而在把 SOP 和中间结构标准化。

  • 可把任务先拆成“目标、参与方、资源、同步条件、互斥条件、时限”六段，再送 LLM。
  • 可增加一个解析后校验器，对 groupCnt、groupMsns.length、groupScope 做结构检查。
  • 可把失败时的最小重试策略写成固定协议，而不是直接 Failed。
阶段 2 · 分组与广播

AssignGroups() 不是交给 LLM 的，而是本地按人数切组，再用消息把任务图发给每个成员

任务解析完成后，真正的分组逻辑由 PlanningModule.AssignGroups() 本地执行。 它根据 groupCnt 把当前注册的 Agent 按顺序切成若干组，再用 SelectLeader() 以“当前电量最高”规则选组长，最后向每个成员发送 GroupBootstrapPayload。这里才是 Agent 间通信第一次进入主链。

AssignGroups() SelectLeader() CommunicationModule.SendScopedMessage() OnGroupBootstrap()

输入

  • ParsedMission 全局任务图
  • CommunicationManager.GetAllAgentIds()
  • 各 Agent 当前电量，用于选组长

输出

  • GroupDef[] groups
  • GroupBootstrapPayload
  • 每个成员本地更新 myGroup、isLeader、props.TeamID

关键字段

  • groupId：形如 g0、g1
  • leaderId：由本地电量最高者决定
  • constraints：只发全局约束或本组约束

涉及函数

  • PlanningModule.AssignGroups()
  • PlanningModule.SelectLeader()
  • CommunicationModule.SendScopedMessage()
  • PlanningModule.OnGroupBootstrap()

上下游关系

  • 上游：LLM#1 已给出全局任务图
  • 中游：通信总线第一次真正承载结构化任务数据
  • 下游：组长进入 LLM#2，成员等待槽位广播

调用链引用

AssignGroups() -> comm.SendScopedMessage(DirectAgent, MessageType.GroupBootstrap, payload) -> CommunicationManager.ProcessMessage() -> CommunicationModule.ReceiveMessage() -> CommunicationModule.ProcessMessages() -> PlanningModule.OnGroupBootstrap()。

字段内嵌 首次出现的数据结构：GroupDef、GroupBootstrapPayload、AgentMessage v
GroupDef.groupId 组标识，白板后续也会按这个分桶存储条目。
GroupDef.mission 这一组接下来真正要完成的子任务文本，是 LLM#2 生成槽位时的核心输入。
GroupBootstrapPayload.constraints 只给“全局约束 + 本组约束”，避免对抗场景里把别组内部协同暴露出来。
AgentMessage.Scope 这里走的是 DirectAgent，不是全局广播。消息内容在 Content 里以 JSON 形式传输。
props.TeamID OnGroupBootstrap() 会把它同步成组序号，因此监控页和组内查询都依赖这里的更新。
审阅观察 这里的“协同约束隔离”已经做了，但组内切分策略仍然非常朴素 v

目前切组是按 Agent 注册顺序和人数均分完成的，而不是按平台类型、角色、位置、剩余电量联合优化。 组长选择也只看电量，不看通信范围、机型、负载或地理位置。 对简单演示够用，但对异构多机或远距离场景会比较粗糙。

对抗任务的广播

红队和蓝队分别收到自己的组任务，例如“围堵”和“撤离”。

优化建议 任务分组可以参考 MRTA 与 CBBA，不必停留在“顺序均分 + 电量最高当组长” v

Gerkey 与 Matarić 的 MRTA taxonomy 给出的是“怎样把任务分配问题形式化”，而 CBBA 给出的是 “分布式任务分配时怎样在局部通信下达成一致”。这两类工作都比当前的顺序切组规则更适合异构无人机系统。

  • MRTA 视角：先定义任务需求、平台能力、代价函数，再分组。
  • CBBA 视角：把局部出价和共识机制放进组形成或槽位冲突消解阶段。
  • 最小落地路径：先不改 UI，只给 SelectLeader() 和分组器引入更丰富的评分函数。
博弈论插入点 若你想把博弈论前移，第一站可以放在“组形成”或“队伍角色竞争”上 v

这一段最适合引入的是“合作博弈 / 竞拍博弈”思想，而不是对抗博弈。 当多个平台都希望拿到更高收益任务时，可以把组形成视为带约束的效用最大化问题。

  • 合作任务：做能力-任务匹配博弈，决定谁该进哪个子组。
  • 竞争任务：做竞拍或 contract-net 风格的角色认领。
  • 对抗任务：队内仍可先做角色分配博弈，再进入对手博弈。
阶段 3 · LLM#2 槽位生成

进入组内之后，组长先用 RunLLM2() 生成 PlanSlot[]，槽位是“个人职责模板”

组长拿到本组任务 myGroup.mission、成员数和组内约束后，会调用 RunLLM2() 生成长度严格等于组成员数的 PlanSlot[]。 槽位里包含角色、职责描述、完成条件以及它关联的约束 ID。注意这里仍然不是动作序列，而是 “给某一位成员看的完整职责槽”。

RunLLM2() PlanSlot[] constraintIds[] MessageType.SlotBroadcast

输入

  • myGroup.mission：本组子任务文本
  • memberCount：组内成员数
  • _constraintDict：本组可见约束全集

输出

  • PlanSlot.slotId / role / desc / doneCond
  • 槽位广播给成员
  • 组长自己也会立刻进入 LLM#3 选槽

关键字段

  • desc：个人完整职责描述
  • constraintIds：槽位绑定的约束引用
  • role：从 RoleType 枚举中选

涉及函数

  • PlanningModule.RunLLM2()
  • CommunicationModule.SendScopedMessage()
  • PlanningModule.OnSlotBroadcast()

上下游关系

  • 上游：组信息已经确定
  • 中游：责任模板从组层落到成员候选层
  • 下游：所有成员开始竞争或认领槽位
字段内嵌 首次出现的数据结构：PlanSlot 与 SlotBroadcastPayload v
PlanSlot.slotId 槽位唯一 ID，例如 s0、s1。后续选槽和确认都只引用这个 ID。
PlanSlot.desc 个人职责的自然语言描述，是 LLM#4 拆步骤时的直接输入。
PlanSlot.doneCond 槽位完成条件。不是动作层条件，而是成员职责层条件。
PlanSlot.constraintIds 只是引用，不复制约束内容。真正查询时仍回到 PlanningModule.GetConstraint()。
SlotBroadcastPayload.slots 组长把整组槽位列表发送给每个成员，让成员自己做适配选择。
审阅观察 LLM#2 已经承载了大部分“角色分工”责任，因此它的输出质量直接决定后续是否混乱 v

当前代码没有给槽位生成增加后验结构检查，例如： constraintIds 是否真的存在、同类成员的 desc 是否重复、 是否把 C1 的独占资源错误分给两个槽位。这意味着一旦 LLM#2 在这里出错，后面的所有成员都会在错误的槽位池里选。

当前流程是“组长先生成槽位，再让成员自己选”，本质上已经很接近 Contract Net 的 “招标 - 投标 - 授标”形态，只是投标和授标都还比较弱。

  • Contract Net 可把当前槽位广播改成标准化招标，成员回报成本、位置、剩余电量等。
  • MRTA 可让槽位生成前就考虑平台异构性和资源约束。
  • CBBA 则适合进一步处理多轮冲突与共识收敛，而不只是“先到先得”。
博弈论插入点 如果要把“博弈论相关”真正加进系统，这一段就是第一个最自然的落点 v

组长产槽之后，成员选槽之前，其实就已经形成了一个带约束的策略空间： 每个成员选择哪一个槽位，不只是“自己最喜欢哪个”，还要看别人会不会选同一个。

  • 可把槽位选择写成简单拍卖博弈：报价函数由距离、电量、角色匹配度组成。
  • 可把组长分配写成 Stackelberg 风格：组长先出机制，成员再响应。
  • 可把重复冲突看成潜在博弈，寻找更稳定的槽位均衡分配。
阶段 4 · 选槽与确认

每个成员用 RunLLM3() 选槽，组长再用 ResolveAndConfirm() 做唯一分配

成员收到 SlotBroadcastPayload 后，会根据自己的当前位置、电量和槽位列表调用 LLM#3 输出一个 slotId。成员把结果以 SlotSelectPayload 发给组长。 当组长收齐所有选择后，由 ResolveAndConfirm() 依据时间戳做“先到先得”分配， 再给每个成员发 SlotConfirmPayload 和 StartExecPayload。

  • 组长维护的 selectionTs 时间戳

输出

  • SlotSelectPayload：成员想要哪个槽位
  • SlotConfirmPayload：最终分配给谁
  • StartExecPayload：允许进入步骤拆解

关键字段

  • agentId：谁在选择或被确认
  • slotId：成员选择的槽位 ID
  • confirmedSlot：成员本地最终认领的槽位

涉及函数

  • PlanningModule.OnSlotConfirm()
  • PlanningModule.OnStartExec()

  • 中游：职责模板被唯一绑定到具体成员
  • 下游：成员拿着自己的职责进入 LLM#4 步骤拆解
字段内嵌 首次出现的数据结构：SlotSelectPayload、SlotConfirmPayload、StartExecPayload v

审阅风险 ResolveAndConfirm() 当前是“先到先得”，这对冲突处理来说过于简单 v

当前组长冲突消解逻辑是：组长自己优先，其余成员按选择消息到达时间排序； 如果某人想要的槽位已被占用，就直接给剩余槽位中的第一个。这种方法实现简单， 但它没有显式使用角色匹配度、距离代价、平台能力或历史可靠性。

  • 合作任务中：可能把更适合某区域的成员挤去次优槽位。
  • 竞争任务中：可能把高价值槽位分给消息更快的人，而不是更优执行者。
  • 对抗任务中：可能让队内追击/封堵角色分配不稳定。

西区更近、电量更高的成员通常会偏向选西区槽位。

竞争任务的认领

选槽输入
同一组内可能都想选“抢资源点 A”的高收益槽位。
现有机制
冲突由先到先得处理，不是拍卖、也不是效用最大化。
后果
槽位分配容易受消息延迟影响，而不一定对应更优策略。
对抗

对抗任务的认领

红队示例
红队两个成员都想拿“主追击”槽位，组长必须做冲突裁决。
现有机制
依旧是先到先得，这会让对抗队内角色分工不够稳定。
建议方向
应把地理位置、拦截角度、速度优势纳入裁决，而不是只看消息时间。
优化建议 这一段最适合落地 Contract Net / 竞价分配 / CBBA 共识机制 v

如果要把当前槽位冲突处理做成真正可扩展的协同约束实现， ResolveAndConfirm() 是最应该升级的节点。

  • Contract Net：成员不只报 slotId，还报代价、预计完成时间、风险。
  • CBBA：冲突槽位通过局部出价和共识迭代收敛，而不是一次性裁决。
  • MRTA：可把槽位价值函数统一成平台能力与任务收益的映射。
博弈论插入点 你提到的“博弈论还没做”，当前最适合加在这里 v

这里就是一个天然的资源分配博弈：多个成员争有限槽位，且每个槽位的收益并不相同。

  • 若组长先公布规则、成员再响应，可建成 Stackelberg 风格分配机制。
  • 若成员同步报价并竞争，可建成竞拍博弈或机制设计问题。
  • 若希望得到稳定分配，可考虑潜在博弈或匹配博弈视角。
所以“博弈论可以加在哪里”的第一个答案不是运动执行层，而就是 RunLLM3() + ResolveAndConfirm() 这一对函数之间。
阶段 5 · LLM#4 步骤拆解

槽位还不是动作；RunLLM4() 会把 confirmedSlot.desc 拆成 PlanStep[]

成员一旦拿到自己被确认的槽位，并收到 StartExecPayload，就会调用 RunLLM4()。这里的职责是把个人职责文本拆成有序步骤，并把槽位级约束重新分配到合适步骤上。 从这里开始，系统第一次具备“步骤级执行单元”。但它依然不是原子动作，原子动作还要到 ADM 再生成。

RunLLM4() PlanStep[] AgentPlan PlanningState.Active

输入

  • confirmedSlot.desc
  • confirmedSlot.doneCond
  • confirmedSlot.constraintIds 对应的约束对象
  • 当前位置与电量

输出

  • PlanStep.stepId / text / doneCond / constraintIds
  • AgentPlan 被写入本地
  • 规划状态进入 Active

关键字段

  • stepId：例如 step_1
  • constraintIds：把约束绑定到具体步骤，而非整个槽位
  • curIdx：当前执行到哪一步

涉及函数

  • PlanningModule.RunLLM4()
  • PlanningModule.GetConstraint()
  • PlanningModule.HasActiveMission()
  • PlanningModule.GetCurrentStep()

上下游关系

  • 上游：槽位已唯一确认
  • 中游：个人职责被切成步骤序列
  • 下游：ADM 只消费当前 PlanStep
字段内嵌 首次出现的数据结构：PlanStep 与 AgentPlan v
PlanStep.text 给 ADM 继续解释的自然语言步骤文本，例如“等 A 就绪后飞往 B 区”。
PlanStep.doneCond 步骤完成条件，供 ADM 决定何时把该步骤判定为完成。
PlanStep.constraintIds 关键桥梁字段。之后白板、裁判、同步、互斥全都依赖它把约束接入当前步骤。
AgentPlan.curIdx 当前步骤游标。每次 CompleteCurrentStep() 递增，最终走到 Done。
审阅观察 LLM#4 负责“把约束分配到步骤”，因此一旦绑定错，后续白板和同步都会错位 v

当前步骤拆解虽然给了比较明确的规则，例如 C1 绑首个进入目标的步骤、C2 绑最后一个实质性动作步骤、 C3 sign=+1 绑第一步、C3 sign=-1 绑共享资源步骤，但它依然完全依赖 LLM 输出。 项目里没有后验检查器去验证某条约束是否真的被绑定到了合理的步骤上。

阶段 6 · ADM 进入执行

ActionDecisionModule.StartStep() 先建立执行上下文，再把当前步骤交给滚动规划循环

当 PlanningModule.HasActiveMission() 为真，且当前 ADM 处于可接受新步骤的空闲态时， IntelligentAgent.MakeDecisionCoroutine() 会进入 ActionDecisionModule.DecideNextAction()。这里先通过 PlanningModule.GetCurrentStep() 拿到当前步骤，再调用 StartStep(step) 构造 ActionExecutionContext。 这个上下文是后续滚动决策的工作内存。

输入

  • 当前角色、位置、任务 ID
  • 通过 constraintIds 解出的步骤级约束对象

输出

  • ActionExecutionContext ctx
  • stepConstraints：步骤关心的结构化约束对象集合
  • 状态进入 ADMStatus.Interpreting

关键字段

  • ctx.stepText：后续滚动提示词的主任务句
  • ctx.executedActionsSummary：滚动规划历史
  • ctx.iterationCount：本步骤已经滚动多少轮

涉及函数

  • ActionDecisionModule.DecideNextAction()
  • ActionDecisionModule.StartStep()
  • PlanningModule.GetCurrentStep()
  • PlanningModule.GetConstraint()

上下游关系

  • 上游：步骤序列已经就绪
  • 中游：构造执行上下文，把“计划层”切到“决策层”
  • 下游：进入真正的滚动 LLM 决策
字段内嵌 首次出现的数据结构：ActionExecutionContext v
ctx.stepConstraints 本步骤真正会读取的约束对象集合。白板、裁判、等待、同步都只看这里。
ctx.currentLocationName 步骤起始位置名，用于给滚动提示词增加空间上下文。

ctx.executedActionsSummary 已执行原子动作摘要。让 LLM 不只看当前环境，也看本步骤已走到哪里。

这一点对理解白板和裁判非常重要：它们不是“全局解释器”，而是被 ADM 在当前步骤上下文里读取的局部信号源。 所以你在主流程里看到白板、裁判、感知，并不是并列模块，而是都被折进了 ActionExecutionContext 服务当前步骤。

阶段 7 · 滚动决策与协同约束

RunRollingLoop() 才是“步骤变动作序列”的核心：白板、裁判、地图、感知、LLM 都在这里汇流

当前系统最重要的真实主链节点就是 ActionDecisionModule.RunRollingLoop(PlanStep step)。 每一轮它先读白板上下文，再检查 C3 等待，再读裁判事件，再拼装局部地图和感知信息， 然后构造滚动提示词，调用 LLM 返回 RollingPlanResult { isDone, doneReason, nextActions[] }。 如果步骤未完成，则把这一批 AtomicAction[] 交给执行层； 如果步骤已完成，则写白板完成信号并推进到下一步。

ReadWhiteboardContext() CheckC3WaitConditions() ReadRefereeContext() BuildRollingPrompt() RollingPlanResult

输入

  • 当前 PlanStep 与步骤约束
  • 本地感知快照、当前位置、局部地图
  • 白板条目 WhiteboardEntry
  • 裁判事件 RefereeEvent

输出

  • RollingPlanResult.isDone
  • RollingPlanResult.nextActions
  • 白板信号写入：IntentAnnounce、DoneSignal
  • 必要时推进到下一步或结束任务

关键字段

  • AtomicAction.type：MoveTo、Observe、Wait 等
  • RollingPlanResult.doneReason
  • WhiteboardEntry.entryType：IntentAnnounce / ReadySignal / DoneSignal

涉及函数

  • ActionDecisionModule.RunRollingLoop()
  • BuildRollingPrompt()
  • WriteC3MutexClaims()
  • WriteWhiteboardIntentSignals()
  • WriteWhiteboardDoneSignals()
  • WaitForC2Sync()

上下游关系

  • 上游：步骤上下文已经固定
  • 中游：多源上下文融成单轮 LLM 决策
  • 下游：执行层消耗原子动作；完成后再回到此循环

调用链引用

StartStep() -> RunRollingLoop() -> ReadWhiteboardContext() -> CheckC3WaitConditions() -> ReadRefereeContext() -> BuildRollingPrompt() -> LLMInterface.SendRequest() -> RollingPlanResult -> ctx.actionQueue = nextActions -> AgentMotionExecutor.GetCurrentAction()。

字段内嵌 首次出现的数据结构：RollingPlanResult、AtomicAction、WhiteboardEntry、RefereeEvent v
RollingPlanResult.isDone 当前步骤是否结束。如果为真，则不再把步骤文本继续拆动作，而是直接写同步信号并推进步骤。
AtomicAction.targetName 动作目标名。MoveTo 和互斥占位写白板时最依赖它。
WhiteboardEntry.progress 当前项目里被拿来记录互斥目标名或文本进度。对于 C3 sign=-1，非常关键。
RefereeEvent.payload 裁判广播的客观事实描述。ADM 只会在 ReadRefereeContext() 中按约束过滤并消费它。

已接入 白板在主链中的真实作用：不是独立模块，而是滚动循环读写的组内共享上下文 v

这里的白板已经真正接进主链，而且接得很清楚： ReadWhiteboardContext() 读取和当前步骤约束有关的条目， WriteWhiteboardIntentSignals() 在执行前写意图， WriteC3MutexClaims() 把互斥目标写入 IntentAnnounce.progress， WriteWhiteboardDoneSignals() 在步骤完成时写 DoneSignal。

  • C2：通过 DoneSignal + WaitForC2Sync() 实现组内同步。
  • C3 sign=-1：通过白板占位 + 提示词注入，让 LLM 主动避开已被占的目标。
  • 白板不是替 LLM 决策，而是给 LLM 提供“别人已经在做什么”的上下文。

队内成员仍可以共享“封堵方向”“追击意图”等组内状态。
裁判
按接口设计，应广播“目标已逃脱”“某 Agent 被淘汰”等客观事实，但当前运行链没有真正接满。
动作输出
适合滚动生成“追击、绕行、观察、等待封口”等动作批次。
审阅风险 裁判链路当前是“接口存在但主链未接满”，不能把它当成已闭环实现 v

当前代码确实有 RefereeManager、RefereeEvent 和 ReadRefereeContext()，但在整个项目搜索里，没有看到 RefereeManager.RegisterAgent() 或 BroadcastEvent() 的真实调用落地。 这意味着 ADM 的“读取裁判事件”入口已经写好，但真正把事件送进来的人还没接上。

  • 结论：裁判不是不存在，而是处于半接入状态。
  • 文档表达：只能写“预留读取点 + 主链未闭环”，不能写成“裁判已参与所有竞争/对抗决策”。
审阅风险 C3 正向等待读取侧已实现，但 ReadySignal 当前没有找到真实写入点 v

CheckC3WaitConditions() 会读取 SharedWhiteboard.Instance.HasSignal(... ReadySignal)，说明“我等别人就绪再行动”的消费侧已经写好。 但项目里没有搜到任何地方真正写入 WhiteboardEntryType.ReadySignal。

  • 结果：C3 sign=+1 的理想语义在当前代码里只完成了半边。
  • 表现：Agent 可能永远等不到真正的“就绪”白板信号，只能靠别的逻辑或超时结束。
审阅风险 感知触发重规划还停在 pendingPerceptionEvents 队列和 TODO 上 v

PerceptionModule 已经可以把敌方发现等事件推给 ActionDecisionModule.OnPerceptionEvent()，事件也会进入 pendingPerceptionEvents 队列。但真正消费这些事件并触发 LLM-B 重规划的 HandlePendingPerceptionEvents() 仍是 TODO。

  • 所以“感知变化驱动动作再规划”在接口层已经预埋，但主链还没闭合。
  • 这也是为什么当前主链更像“步骤内滚动决策”，而不是“对突发环境强响应的闭环”。
优化与博弈论 竞争/对抗的真正博弈论落点在这里：资源争夺、对手建模、先后手策略都发生在滚动环里 v

如果要把“博弈论相关”做成真正影响动作输出的功能，阶段 7 是最关键的第二个插入点。 因为这里才第一次同时看到：当前步骤、环境事实、队友状态、对手事实和候选动作。

  • 竞争任务：可用局部 Nash / 潜在博弈思想，让多个成员围绕共享资源做稳定目标选择。
  • 对抗任务：可用 Markov Games 建模多轮追逃；可用 Nash Q-learning 处理一般和局博弈。
  • 有明确先手方的追捕/防逃：可用 Stackelberg 视角建模“先布防，后响应”。
这也是为什么博弈论不该被单独讲成“另一个模块”：它最自然的落点就是 RunRollingLoop() 里“读上下文 -> 生成 nextActions”这一跳。
阶段 8 · 动作执行层

AgentMotionExecutor 消费当前原子动作，把文字动作真正变成移动、巡逻、观察、等待

当前系统把原子动作交给 Execution/AgentMotionExecutor.cs。 它在 Update() 里轮询 adm.GetCurrentAction()，动作变化时启动新的执行协程， 再根据动作类型进入 DoMoveTo()、DoObserve()、 DoWait()、DoPatrol()、DoFormation()、 DoEvade()。动作完成后调用 adm.CompleteCurrentAction()， 于是控制权回到滚动规划主链。

GetCurrentAction() ExecuteAction() DoMoveTo() adm.CompleteCurrentAction()

输入

  • AtomicAction.type、targetName、duration
  • 地图网格 CampusGrid2D
  • Rigidbody 物理参数

输出

  • 物理移动、巡逻、观察、等待等真实执行
  • 动作完成回调到 ADM
  • 批次动作完成后再回到滚动规划

关键字段

  • currentAction：正在执行的动作
  • currentPath / waypointIdx：A* 路径缓存
  • hoverHeight / pdKp / pdKd：执行层物理参数

涉及函数

  • AgentMotionExecutor.PollADM()
  • ExecuteAction()
  • DoMoveTo()
  • DoObserve()
  • ActionDecisionModule.CompleteCurrentAction()

上下游关系

  • 上游：滚动规划已经给出当前批次动作
  • 中游：把动作落成物理行为
  • 下游：动作完成通知 ADM；步骤完成再通知 PlanningModule
主链闭环 从这里开始，动作序列真正变成空间行为；动作完成后又回流到滚动规划 v

这也是“直到变成动作序列的整个流程”最终落地的节点：前面的计划、槽位、步骤、白板和裁判， 最后都会归结为当前批次的 AtomicAction[]。执行层并不理解组任务、也不理解裁判规则， 它只负责把当前动作做完，再把“做完了”这件事实反馈回主链。 

// 执行层的最短闭环：取动作 -> 执行 -> 完成回调
nextAction = adm.GetCurrentAction();
if (nextAction.type == MoveTo) {
  // 先做 A* 路径，再逐路点飞行
  DoMoveTo(nextAction);
}
// 当前动作执行完毕后
adm.CompleteCurrentAction();
审阅观察 FormationHold 的编队偏移仍然硬编码，说明动作层也还存在待补口 v

当前 DoFormation() 使用固定偏移 (3, 0, 0)。 这意味着即使上游 LLM 已经知道应采用更合理的编队几何，动作层也无法精细接收。 它不会阻断主链，但会限制协同动作表现力。

代码审阅结论汇总

R1 · 裁判链半接入

RefereeManager、RefereeEvent 与 ActionDecisionModule.ReadRefereeContext() 已存在，但项目内未搜到 RefereeManager.RegisterAgent() 与 BroadcastEvent() 的实际接线， 因此不能把裁判写成当前已闭环主链。

R2 · C3 正向等待只做了读，没有做写

CheckC3WaitConditions() 会等 ReadySignal， 但项目中未搜到 ReadySignal 的实际写入点，所以 sign=+1 的完整语义没有真正闭环。

R3 · 感知驱动重规划仍是 TODO

pendingPerceptionEvents 队列和 OnPerceptionEvent() 已接入， 但 HandlePendingPerceptionEvents() 仍未实现，导致“看到敌人后重规划”还停在预埋状态。

R4 · 旧约束文档与当前代码不一致

constraint-reference.html 中出现了 C4 Priority、 involvedAgents 等设计字段，但当前 StructuredConstraint 实装中并没有这些字段。

R5 · LLM 密钥暴露

LLMInterface 默认配置中直接包含 API Key，这是已经进入主链的高风险安全问题。

R6 · 槽位冲突处理过于朴素

ResolveAndConfirm() 主要按先到先得处理冲突，没有利用任务收益、角色匹配度、距离代价等更强决策信息。

以上问题都已经在前面的主链节点旁做了嵌入式标注。这里集中列出，只是为了让你在审阅时能快速抓到实现边界。
