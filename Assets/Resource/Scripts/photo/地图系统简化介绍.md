# 地图系统简化介绍

## 1. CampusJsonMapLoader 中的地图要素类型

`CampusJsonMapLoader` 中地图“大节点”本质上不是寻路格点，而是校园地图里的要素对象。代码里用 `CampusFeatureKind` 表示要素类型，当前包括：

- `Building`：建筑
- `Sports`：体育区域
- `Water`：水域
- `Road`：道路
- `Expressway`：干道
- `Bridge`：桥梁
- `Parking`：停车区域
- `Green`：绿地
- `Forest`：林地区域
- `Other`：其他区域

每个地图要素对象除了类型，还会保存一组基础信息：

- `uid`：唯一标识
- `name`：名称
- `tags`：附加属性
- `linePoints`：线型要素的坐标点
- `outerRings / innerRings`：面型要素的外轮廓和内孔洞
- `bounds`：包围盒

因此，`CampusJsonMapLoader` 这一层保存的是“地图要素对象”，不是“网格节点”。

## 2. CampusGrid2D 中如何存储这些地图要素

`CampusGrid2D` 会把这些地图要素进一步转成二维逻辑网格。它的核心作用不是显示地图，而是把地图变成“格子化、可查询、可寻路、可按名字定位”的内存结构。

### 2.1 格子级存储

最核心的是几组二维数组：

- `blockedGrid[x,z]`
  表示某个格子是否阻塞，不可通行。

- `cellTypeGrid[x,z]`
  表示某个格子的类型。类型枚举包括：
  `Free / Building / Water / Road / Expressway / Bridge / Parking / Green / Forest / Sports / Other`

- `cellFeatureUidGrid[x,z]`
  记录该格子当前对应的地图要素 `uid`。

- `cellFeatureNameGrid[x,z]`
  记录该格子当前对应的地图要素名称。

也就是说，网格中的每个单元格不仅知道“能不能走”，还知道“自己属于什么地图区域、对应哪个要素对象”。

### 2.2 别名与定位索引

为了让上层模块按名称或别名快速找到目标，`CampusGrid2D` 还维护了几组映射：

- `featureAliasCellMap`
  把运行时别名映射到一个代表格子。

- `featureAliasUidMap`
  把别名映射回原始要素 `uid`。

- `featureAliasNameMap`
  把别名映射回原始要素名称。

这部分主要服务“按文本目标定位地图区域”的需求。

### 2.3 要素级空间画像

除了格子数组，`CampusGrid2D` 还会按地图要素保存更高层的空间摘要，核心是：

- `featureSpatialProfileByUid`

它按 `uid` 保存每个地图要素的空间画像，主要包括：

- 占用了多少格子
- 格子范围边界
- 中心格
- 世界坐标中心
- 锚点格和锚点世界坐标
- 覆盖半径
- 成员实体列表

这意味着上层模块不需要重新遍历整个网格，就能直接拿到某个建筑、绿地、道路区域的大致空间信息。

### 2.4 集合成员关系

另外还有：

- `featureCollectionMembers`

它用来保存一组相关地图要素之间的成员关系，便于后续按集合做组织和查询。

## 3. CampusSmallNodeSpawner 会刷哪些小节点

`CampusSmallNodeSpawner` 刷的是运行时“小节点”，也就是地图上的附加实体，而不是校园底图本身。

代码里的 `SmallNodeType` 当前包括：

- `Unknown`
- `Tree`
- `Pedestrian`
- `Vehicle`
- `ResourcePoint`
- `TemporaryObstacle`
- `Agent`
- `Custom`

其中，当前默认规则主要会刷三类：

- `Tree`：树木
- `Pedestrian`：行人
- `ResourcePoint`：资源点

默认分布关系大致是：

- `Tree` 多刷在 `Forest / Green`
- `Pedestrian` 多刷在 `Road / Green`
- `ResourcePoint` 多刷在 `Parking / Green / Other`

从用途上看，这些小节点比地图大要素更偏“运行时实体”：

- `Tree`、`TemporaryObstacle` 更像障碍物
- `Pedestrian`、`Vehicle`、`Agent` 更像动态实体
- `ResourcePoint` 更像可交互目标

## 4. 小节点在运行时如何存储

小节点的存储方式和地图大要素不同。地图大要素主要存放在 `CampusGrid2D` 的网格和索引中，小节点则主要存放在 Unity 场景对象及其运行时组件里。

### 4.1 生成对象列表

- `spawnedNodes`

这个列表保存当前已经生成出来的所有小节点对象，便于统一清理和管理。

### 4.2 按类型组织层级

- `typeRoots`

它按小节点类型建立场景层级根节点，例如：

- `Map/Tree`
- `Map/Pedestrian`
- `Map/ResourcePoint`

这样做的作用是让场景中的小节点不是混在一起，而是按类型分组管理。

### 4.3 按类型记录编号

- `typeCounters`

它记录每种类型已经生成了多少个节点，用来给对象命名，例如：

- `Tree_1`
- `Tree_2`
- `Pedestrian_1`

### 4.4 每个对象上的运行时语义

每个生成出来的小节点对象上，都会挂一个 `SmallNodeRuntimeInfo` 组件。这个组件保存的是小节点自己的运行时语义信息，包括：

- `nodeType`：节点类型
- `isDynamic`：是否动态
- `blocksMovement`：是否阻挡移动
- `serialIndex`：序号
- `displayName`：显示名
- `sourceSpawner`：来源刷子

这意味着其他系统在拿到一个场景对象后，不必猜它是什么，只要读取这个组件就能知道它属于哪一类小节点。

### 4.5 动态节点的移动状态

如果某个节点属于动态类型，还会额外挂 `CampusSmallNodeWanderMover`。这个组件负责保存和驱动它的移动状态，主要包括：

- 移动范围
- 固定高度
- 移动速度
- 重新选目标的时间间隔
- 当前移动目标点

因此，静态节点和动态节点在运行时的差别也很明确：

- 静态节点：只有对象本体和 `SmallNodeRuntimeInfo`
- 动态节点：除了上述信息，还会额外挂移动组件

## 5. 一句话总结

这个地图系统实际上分成两套存储：

- 地图大要素存储在 `CampusGrid2D` 的网格和索引中，服务于地图语义、定位和寻路。
- 小节点存储在 Unity 场景对象及其运行时组件中，服务于运行时实体表达和交互。

两者配合后，系统既有“可查询的地图”，也有“可运行的场景实体”。

## 6. 验证口径

可以用下面几个现象快速检查理解是否正确：

- 导入地图后，不同区域会被标记为 `Building`、`Water`、`Road`、`Green` 等格子类型。
- 查询某个格子时，可以拿到该格子的类型、所属要素 `uid` 和名称。
- 刷点后，场景层级里能看到按类型分组的小节点对象。
- 动态小节点会额外挂移动组件，静态节点不会。
